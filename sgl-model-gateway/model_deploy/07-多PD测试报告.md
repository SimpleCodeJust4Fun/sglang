## 调度策略测试结果

### 1. Round Robin (轮询)

**原理**: 按顺序轮流分配请求到不同的Prefill-Decode对

| 测试项 | 结果 |
|--------|------|
| 简单请求 | ✓ "Hello! How can I assist you today?" |
| 中文请求 | ✓ "我是由阿里云开发的超大规模语言模型，我叫通义千问。" |
| 并发(5) | ✓ 5/5 成功 |
| 并发耗时 | 961ms |
| Token使用 | prompt=30, completion=10 |

**特点**: 
- 请求均匀分布到各个节点
- 第一轮并发延迟较高（~961ms），可能是Gateway冷启动导致

### 2. Cache Aware (缓存感知)

**原理**: 优先将请求路由到已有相关缓存的节点，减少KV Cache传输

| 测试项 | 结果 |
|--------|------|
| 简单请求 | ✓ "Hello! How can I assist you today?" |
| 中文请求 | ✓ "我是来自阿里云的超大规模语言模型，我叫通义千问。" |
| 并发(5) | ✓ 5/5 成功 |
| 并发耗时 | 302ms |
| Token使用 | prompt=30, completion=10 |

**特点**:
- 并发性能最好（~302ms）
- 利用缓存减少不必要的数据传输

### 3. Random (随机)

**原理**: 随机选择节点分配请求

| 测试项 | 结果 |
|--------|------|
| 简单请求 | ✓ "Hello! How can I assist you today! Please let me know..." |
| 中文请求 | ✓ "你好，我是来自阿里云的大规模语言模型，我叫通义千问。" |
| 并发(5) | ✓ 5/5 成功 |
| 并发耗时 | 301ms |
| Token使用 | prompt=30, completion=30 |

**特点**:
- 并发性能与cache_aware相当（~301ms）
- 请求分布不可预测

## 关键发现

### 显存限制
1. **16GB显存无法运行两个7B实例** - 即使mem-fraction低至0.2也会OOM
2. **16GB显存可以运行四个0.5B实例** - mem-fraction=0.15, context=2048
3. 显存使用率高达92%，余量很小

### 调度策略
1. **所有三种策略都正常工作** - round_robin, cache_aware, random
2. **cache_aware和random并发性能相当** - 约300ms
3. **round_robin首次延迟较高** - 可能是冷启动效应

### 多PD架构
1. **2P+2D架构成功验证** - 所有服务健康，请求正常路由
2. **KV Cache传输正常** - Prefill的bootstrap端口正确工作
3. **Gateway调度正确** - 请求被分发到不同的Prefill-Decode对

## 故障排查

### 问题1: Decode Server OOM (mem-fraction=0.22)
**症状**: 进程被kill，退出码137
**原因**: 两个Prefill已占用10.5GB，剩余5.5GB不够两个Decode实例
**解决**: 降低mem-fraction到0.15，减少context-length到2048

### 问题2: 脚本被pkill杀死
**症状**: 启动脚本退出码15
**原因**: pkill会匹配脚本自身进程
**解决**: 使用`(killall ...)` 子shell或`killall`替代`pkill`
