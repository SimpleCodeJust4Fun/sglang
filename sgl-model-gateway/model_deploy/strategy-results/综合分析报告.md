# PD调度策略综合分析报告

> **测试日期**: 2026-04-24  
> **测试环境**: WSL2 + RTX 4070Ti Super + CUDA 12.8  
> **模型**: Qwen2.5 系列  
> **架构**: Prefill-Decode 分离架构 (2P + 2D)

---

## 1. 测试概述

### 1.1 测试目标

验证 SGLang Model Gateway 的 8 种 PD（Prefill-Decode）调度策略在真实环境中的路由行为和性能表现。

### 1.2 测试配置

| 配置项 | 值 |
|--------|-----|
| Prefill Worker 数 | 2 (ports: 30000, 30001) |
| Decode Worker 数 | 2 (ports: 30010, 30011) |
| Bootstrap Server | 127.0.0.1:9000, 127.0.0.1:9001 |
| 每策略请求数 | 100 |
| 总请求数 | 800 |
| Prompt 类型 | 多样化 (平均 46.15 tokens) |
| max_new_tokens | 50 |
| temperature | 0.7 |

### 1.3 测试指标

- **成功率**: 请求是否成功完成 (HTTP 200)
- **Worker 分布**: Prefill/Decode Worker 的负载分配
- **Worker 配对**: Prefill→Decode 的组合分布
- **响应延迟**: 端到端延迟 (平均/最小/最大)
- **KV Cache 利用率**: 缓存命中率

## 2. 策略分析详情

### 2.1 round_robin (轮询策略)

**状态**: ✅ PASS | **成功率**: 100% | **平均延迟**: 0.297s

#### 调度分布

| Worker | Prefill 分布 | Decode 分布 |
|--------|-------------|-------------|
| Worker-1 | 50 (50%) | 50 (50%) |
| Worker-2 | 50 (50%) | 50 (50%) |

#### Worker 配对

| 配对 | 数量 | 占比 |
|------|------|------|
| prefill-1 → decode-2 | 50 | 50% |
| prefill-2 → decode-1 | 50 | 50% |

#### 分析

- **完美的 50/50 分布**: 轮询策略实现了数学上完美的负载均衡
- **反亲和性配对**: prefill-N 始终与 decode-(3-N) 配对，避免同索引配对
- **确定性行为**: 奇数请求走 prefill-1→decode-2，偶数请求走 prefill-2→decode-1
- **延迟稳定**: 大部分请求集中在 310-320ms

**结论**: 轮询策略工作正常，适合需要严格负载均衡的场景。

---

### 2.2 random (随机策略)

**状态**: ✅ PASS | **成功率**: 100% | **平均延迟**: 0.297s

#### 调度分布

| Worker | Prefill 分布 | Decode 分布 |
|--------|-------------|-------------|
| Worker-1 | 50 (50%) | 50 (50%) |
| Worker-2 | 50 (50%) | 50 (50%) |

#### Worker 配对

| 配对 | 数量 | 占比 |
|------|------|------|
| prefill-1 → decode-1 | 28 | 28% |
| prefill-2 → decode-2 | 28 | 28% |
| prefill-1 → decode-2 | 22 | 22% |
| prefill-2 → decode-1 | 22 | 22% |

#### 分析

- **均匀随机**: 4 种配对组合都有分布，接近理想的 25% 均匀分布
- **独立选择**: Prefill 和 Decode Worker 的选择相互独立
- **自然方差**: ±3 的偏差在二项分布的标准差范围内 (σ≈4.33)
- **无负载感知**: 不考虑当前 Worker 负载

**结论**: 随机策略工作正常，适合作为基线对比，但无法优化性能。

---

### 2.3 power_of_two (随机二选一)

**状态**: ✅ PASS | **成功率**: 100% | **平均延迟**: 0.299s

#### 调度分布

| Worker | Prefill 分布 | Decode 分布 |
|--------|-------------|-------------|
| Worker-1 | 40 (40%) | 37 (37%) |
| Worker-2 | 60 (60%) | 63 (63%) |

#### Worker 配对

| 配对 | 数量 | 占比 |
|------|------|------|
| prefill-2 → decode-2 | 48 | 48% |
| prefill-1 → decode-1 | 25 | 25% |
| prefill-1 → decode-2 | 15 | 15% |
| prefill-2 → decode-1 | 12 | 12% |

#### 分析

- **负载不均衡**: prefill-2 和 decode-2 承载了更多流量 (60%/63%)
- **对角线偏好**: 同索引配对 (prefill-N→decode-N) 占 73%
- **随机采样特性**: 在小样本 (100) 下可能出现偏差
- **负载感知逻辑**: 选择随机 2 个中负载较低的，但可能导致热点聚集

**结论**: 策略工作正常，但在小样本下负载均衡效果不如轮询。适合大规模并发场景。

### 2.4 cache_aware (缓存感知策略)

**状态**: ✅ PASS | **成功率**: 100% | **平均延迟**: 0.299s

#### 调度分布

| Worker | Prefill 分布 | Decode 分布 |
|--------|-------------|-------------|
| Worker-1 | 52 (52%) | 47 (47%) |
| Worker-2 | 48 (48%) | 53 (53%) |

#### Worker 配对

| 配对 | 数量 | 占比 |
|------|------|------|
| prefill-1 → decode-2 | 28 | 28% |
| prefill-2 → decode-2 | 25 | 25% |
| prefill-1 → decode-1 | 24 | 24% |
| prefill-2 → decode-1 | 23 | 23% |

#### 分析

- **均衡分布**: 48-52% 的分布非常均匀
- **高缓存命中率**: `prefill_cached_token` ≈ `prompt_tokens - 1`，几乎所有 prompt token 都命中缓存
- **仅 1 个新 token**: 每次 prefill 只需处理 1 个新 token
- **所有配对均匀**: 4 种配对都有 23-28 个请求

**结论**: 缓存感知策略工作优秀，大幅减少 prefill 计算量。是生产环境推荐策略。

### 2.5 prefix_hash (前缀哈希策略)

**状态**: ❌ FAIL | **成功率**: 0% | **平均延迟**: 0.422s (错误响应延迟)

#### 调度分布

| Worker | Prefill 分布 | Decode 分布 |
|--------|-------------|-------------|
| unknown | 100 (100%) | 100 (100%) |

#### Worker 配对

| 配对 | 数量 | 占比 |
|------|------|------|
| unknown → unknown | 100 | 100% |

#### 错误信息

```
No available servers: Policy prefix_hash failed to select a prefill worker
```

#### 分析

- **完全失败**: 所有 100 个请求都返回 503 错误
- **Worker 未注册**: 策略无法发现可用的 Prefill Worker
- **可能原因**:
  1. Worker 健康检查失败
  2. 前缀哈希计算逻辑有问题
  3. Worker 注册到路由表时元数据不完整
- **后端日志存在**: 底层 Worker 实际处理了请求，但 Gateway 层无法路由

**结论**: ⚠️ **需要修复**。前缀哈希策略在当前配置下无法工作，建议检查 Worker 注册和健康检查逻辑。

#### Worker 配对

#### 分析

## 7. 结论

本次测试验证了 SGLang Model Gateway 的 8 种 PD 调度策略。其中：

- **6 种策略**功能正常 (成功率 100%)
- **1 种策略**需要修复 (prefix_hash: 100% 失败)

**cache_aware** 策略在生产环境中表现最佳，兼顾了负载均衡和缓存利用率。

**round_robin** 策略在负载均衡方面表现最稳定，适合需要严格均匀分配的场景。
