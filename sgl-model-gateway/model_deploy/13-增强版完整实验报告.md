### 性能分解

| 阶段 | 耗时 | 占比 | 组件 |
|------|------|------|------|
| **Prefill** (KV计算+Bootstrap) | ~37ms | 10.6% | Prefill-2 |
| **Decode** (token生成) | ~300ms | 86.4% | Decode-2 |
| **Gateway** (路由+合并) | ~5ms | 1.4% | Gateway |
| **网络传输** | ~5ms | 1.4% | Client↔Gateway |
| **总计** | **~347ms** | **100%** | 端到端 |

**关键发现**:
1. **Decode占主导** (86.4%) - 自回归生成的特性
2. **Prefill很快速** (10.6%) - 并行计算优势
3. **Gateway开销极小** (1.4%) - Rust高性能
4. **Bootstrap高效** (~8ms传输108KB)

## 第三部分：响应流分析

### 响应路径

**正确的响应流** (基于日志验证):

```
Decode-2生成响应
    ↓
直接返回给Gateway (HTTP 200)
    ↓
Gateway合并响应（如果需要logprobs）
    ↓
返回给Client

注意: 响应不经过Prefill！
```

**日志证据**:

1. **Decode-2日志**:
   ```
   [01:54:10] POST /v1/chat/completions HTTP/1.1" 200 OK
   ← Decode返回200，包含完整响应
   ```

2. **Gateway日志**:
   ```
   [17:54:10] finished processing request
              status_code: 200
              latency: 338736
   ← Gateway收到Decode响应后返回给Client
   ```

3. **Prefill-2日志**:
   ```
   [01:54:10] POST /v1/chat/completions HTTP/1.1" 200 OK
   ← Prefill只返回200确认（不含响应内容）
   ```

**结论**: 响应流是 **Decode → Gateway → Client**，不是 P→D→P→Client。
