## 一、策略分类体系

SGLang Model Gateway 的请求调度策略分为 **4 大类**,共 **9 种 CLI 可用策略** (+ 2 种代码内置但未暴露 CLI):

```
┌─────────────────────────────────────────────────────────────┐
│                   请求调度策略分类体系                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 无状态负载均衡 (3 种)                                      │
│     ├─ RoundRobin          - 轮询                             │
│     ├─ Random              - 随机                             │
│     └─ PowerOfTwo          - 随机二选一                        │
│                                                             │
│  2. 缓存/前缀感知 (2 种)                                       │
│     ├─ CacheAware          - KV Cache 感知 (Radix Tree)       │
│     └─ PrefixHash          - 前缀哈希匹配 (轻量级)              │
│                                                             │
│  3. 会话/手动路由 (1 种)                                       │
│     └─ Manual              - 手动指定 (Routing Key 亲和)      │
│                                                             │
│  4. 异构 GPU 调度 (3 种) ← 新增                                │
│     ├─ RequestSizeBucket   - 请求长度分桶                      │
│     ├─ PerformanceAware    - 性能感知                         │
│     └─ RequestClassification - 请求分类                       │
│                                                             │
│  ── 代码内置但未暴露 CLI (2 种) ──────────────────────────────── │
│     ├─ ConsistentHashing   - 一致性哈希 (仅配置文件可用)        │
│     └─ Bucket              - 自适应分桶 (仅配置文件可用)        │
│                                                             │
│  ── 早期文档中虚构的策略 (代码中不存在) ───────────────────────── │
│     ├─ LeastLoad           - 最小负载 (由 PowerOfTwo 替代)     │
│     ├─ CacheLocality       - 缓存局部性 (由 CacheAware 涵盖)   │
│     ├─ PrefixCache         - 前缀缓存 (实际为 PrefixHash)      │
│     ├─ SessionAffinity     - 会话绑定 (由 Manual 策略替代)      │
│     └─ PDScheduler         - PD 分离调度 (不作为独立策略)       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

> **注意**: 早期版本本文档描述了 12 种策略 (含 LeastLoad、CacheLocality、PrefixCache、SessionAffinity、PDScheduler),经代码核实这些策略在当前代码库中不存在或不作为独立策略暴露。详见各章节的 ⚠️ 标注。

### 分类维度

| 维度 | 说明 | 适用场景 | 代表策略 |
|------|------|---------|----------|
| **无状态** | 不依赖历史状态,决策简单快速 | 同构 GPU 集群,请求独立 | RoundRobin, Random, PowerOfTwo |
| **缓存/前缀感知** | 利用 KV Cache 或前缀哈希优化路由 | 长上下文,重复前缀多 | CacheAware, PrefixHash |
| **手动路由** | 通过 Header 指定目标,提供会话亲和 | 多轮对话,流式交互,调试 | Manual |
| **异构调度** | 适配不同 GPU 特性,智能路由 | H800+H20 混部,异构环境 | RequestSizeBucket, PerformanceAware, RequestClassification |

## 二、无状态负载均衡策略

### 2.1 RoundRobin (轮询)

#### 基本原理

按固定顺序循环分配请求到 Worker,确保每个 Worker 获得均等的请求数。

#### 决策流程

```
请求到达
  ↓
获取 Worker 列表 [W1, W2, W3, ... Wn]
  ↓
current_index = (current_index + 1) % n
  ↓
返回 Worker[current_index]
```

#### 伪代码

```rust
struct RoundRobin {
    current_index: AtomicUsize,
    worker_count: usize,
}

impl LoadBalancingPolicy for RoundRobin {
    fn select_worker(&self, request: &Request) -> Option<WorkerId> {
        let idx = self.current_index.fetch_add(1, Ordering::SeqCst) % self.worker_count;
        Some(WorkerId(idx))
    }
}
```

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| 同构 GPU 集群 | ★★★★★ |
| 请求长度均匀分布 | ★★★★☆ |
| Worker 性能一致 | ★★★★★ |
| 异构 GPU 环境 | ★☆☆☆☆ |

#### 优缺点

**优点**:
- 实现简单,性能开销极小
- 保证长期负载均衡
- 无需维护状态

**缺点**:
- 忽略请求长度差异,可能导致实际负载不均
- 不适用于异构环境
- 无法利用缓存

#### 配置示例

```json
{
  "policy": {
    "type": "round_robin"
  }
}
```

### 2.2 Random (随机)

#### 基本原理

从可用 Worker 中随机选择一个,概率均等。

#### 伪代码

```rust
impl LoadBalancingPolicy for Random {
    fn select_worker(&self, request: &Request) -> Option<WorkerId> {
        let idx = thread_rng().gen_range(0..self.worker_count);
        Some(WorkerId(idx))
    }
}
```

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| 大规模集群 (N > 100) | ★★★★☆ |
| 请求随机到达 | ★★★★★ |
| 避免同步振荡 | ★★★★☆ |

#### 优缺点

**优点**:
- 实现简单
- 大数定律保证长期均衡
- 避免同步请求导致的瞬时负载峰值

**缺点**:
- 短期可能负载不均
- 不感知负载状态
- 不适合小规模集群

### 2.3 LeastLoad (最小负载) ⚠️ 不存在

> **⚠️ 重要**: `LeastLoad` 策略在当前代码库中**不存在**。该策略的"最小负载"功能由 `PowerOfTwo` (随机二选一) 策略替代——PowerOfTwo 随机选2个Worker后取负载较小的，效果接近LeastLoad但避免了全局遍历O(N)的开销。
>
> 如果你需要"选择负载最小的Worker"语义，请使用 `power_of_two` 策略。

#### 原理说明 (仅供理解概念)

选择当前负载最小的 Worker,动态响应实际负载状态。

```
请求到达
  ↓
查询所有 Worker 当前负载 (活跃请求数)   ← 需要 O(N) 全局遍历
  ↓
选择负载最小的 Worker
  ↓
返回 WorkerId
```

#### 为什么不实现 LeastLoad?

| 问题 | PowerOfTwo 如何解决 |
|------|-------------------|
| 全局遍历 O(N) 开销 | 随机选2个，O(1) 开销 |
| 羊群效应 (多请求同时选同一Worker) | 随机分散，降低羊群效应 |
| 大规模集群性能差 | 规模无关，始终 O(1) |

> **理论保证**: PowerOfTwo 的最大负载期望值为 `log log N / log 2`，远优于 Random 的 `O(√N)`，且性能开销仅为 O(1)。

### 2.4 PowerOfTwo (随机二选一)

#### 基本原理

随机选择 2 个 Worker,从中选择负载较小的一个。是 Random 和 LeastLoad 的折中。

#### 决策流程

```
请求到达
  ↓
随机选择 2 个 Worker
  ↓
比较两者的负载
  ↓
返回负载较小的 Worker
```

#### 伪代码

```rust
impl LoadBalancingPolicy for PowerOfTwo {
    fn select_worker(&self, request: &Request) -> Option<WorkerId> {
        let idx1 = thread_rng().gen_range(0..self.worker_count);
        let idx2 = loop {
            let idx = thread_rng().gen_range(0..self.worker_count);
            if idx != idx1 { break idx; }
        };
        
        let w1 = &self.workers[idx1];
        let w2 = &self.workers[idx2];
        
        if w1.active_requests < w2.active_requests {
            Some(w1.id)
        } else {
            Some(w2.id)
        }
    }
}
```

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| 大规模集群 | ★★★★★ |
| 需要负载均衡但开销敏感 | ★★★★★ |
| 请求长度有差异 | ★★★★☆ |

#### 优缺点

**优点**:
- 性能开销远小于 LeastLoad (O(1) vs O(N))
- 负载均衡效果接近 LeastLoad
- 避免"羊群效应"

**缺点**:
- 不是全局最优,是局部最优
- 仍需查询负载状态

#### 理论保证

**定理**: PowerOfTwo 策略的最大负载期望值为 `log log N / log 2`,远优于 Random 的 `O(√N)`。

## 三、缓存感知策略

### 3.1 CacheAware (KV Cache 感知)

#### 基本原理

优先选择已缓存请求 KV Cache 的 Worker,减少重复计算。

#### 决策流程

```
请求到达
  ↓
计算请求的 Cache Key (如 prompt hash)
  ↓
查询各 Worker 的 Cache 命中情况
  ↓
if 有 Worker 命中:
    选择命中率最高的 Worker
else:
    选择负载最小的 Worker
```

#### 伪代码

```rust
impl LoadBalancingPolicy for CacheAware {
    fn select_worker(&self, request: &Request) -> Option<WorkerId> {
        let cache_key = compute_cache_key(&request.prompt);
        
        // 检查缓存命中
        let hit_workers: Vec<_> = self.workers
            .iter()
            .filter(|w| w.cache.contains(cache_key))
            .collect();
        
        if !hit_workers.is_empty() {
            // 选择缓存命中率最高的
            hit_workers
                .iter()
                .max_by(|a, b| a.cache_hit_rate.cmp(&b.cache_hit_rate))
                .map(|w| w.id)
        } else {
            // 缓存未命中,选择负载最小的
            self.least_load_policy.select_worker(request)
        }
    }
}
```

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| 重复 prompt 多 | ★★★★★ |
| 长上下文 | ★★★★★ |
| 批量处理相似请求 | ★★★★☆ |

#### 优缺点

**优点**:
- 大幅减少重复计算
- 降低 TTFT (Time To First Token)
- 节省 GPU 算力

**缺点**:
- 需要维护缓存索引
- 缓存过期策略复杂
- 如果缓存命中率低,效果不明显

#### 调优建议

- **缓存淘汰策略**: LRU (Least Recently Used) 或 LFU (Least Frequently Used)
- **缓存大小**: 根据 GPU 显存调整,一般占用 20-30% 显存
- **命中阈值**: 设置最低命中率,低于阈值不使用缓存

### 3.2 CacheLocality (缓存局部性优化) ⚠️ 不存在

> **⚠️ 重要**: `CacheLocality` 策略在当前代码库中**不存在**作为独立策略。其"缓存局部性优化"功能已由 `CacheAware` 策略的 Radix Tree 前缀匹配机制涵盖。CacheAware 策略会自动追踪每个Worker上已缓存的prompt前缀,并优先路由到有最长前缀匹配的Worker。
>
> 如果你需要缓存感知路由,请使用 `cache_aware` 策略。

#### 概念说明

CacheLocality 的核心思想是: 如果请求的前缀与某个Worker上已有的KV Cache Block匹配,则优先路由到该Worker,以利用连续Block的局部性。当前 CacheAware 策略的 Radix Tree 已实现类似效果。

### 3.3 PrefixHash (前缀哈希匹配)

> **名称更正**: 早期文档称此策略为 "PrefixCache",实际代码中的名称为 `prefix_hash`。

#### 基本原理

PrefixHash 是 CacheAware 的轻量级替代方案。使用一致性哈希环将请求按前缀token hash值路由到对应Worker,实现O(log n)查找而非O(prefix_len)的Radix Tree遍历。

#### 决策流程

```
请求到达
  ↓
提取 prompt 前缀的 N 个 token (默认 256)
  ↓
计算前缀 Hash
  ↓
在一致性哈希环上查找对应 Worker
  ↓
if Worker 负载过高 (load > avg * load_factor):
    沿环查找下一个 Worker
  ↓
返回 Worker
```

#### 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `prefix_token_count` | 256 | 用于哈希的前缀 token 数 |
| `load_factor` | 1.25 | 负载因子阈值,超过时沿环查找下一个 Worker |

#### 配置示例

```json
{
  "policy": {
    "type": "prefix_hash",
    "prefix_token_count": 256,
    "load_factor": 1.25
  }
}
```

#### CacheAware vs PrefixHash 对比

| 维度 | CacheAware | PrefixHash |
|------|-----------|------------|
| **数据结构** | Radix Tree (前缀树) | 一致性哈希环 |
| **查找复杂度** | O(prefix_len) | O(log n) |
| **内存开销** | 中等 (树节点) | 极低 (环节点) |
| **匹配精度** | 最长前缀精确匹配 | Hash 近似匹配 |
| **适用规模** | 中小规模 | 大规模 |
| **Mesh 同步** | 支持 | 不支持 |

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| System Prompt 相同 | ★★★★★ |
| Few-shot 示例复用 | ★★★★☆ |
| 大规模集群 (N > 100) | ★★★★★ |
| 需要轻量级前缀路由 | ★★★★★ |

## 四、会话/手动路由策略

### 4.1 Manual (手动指定路由) ← 项目原有

#### 基本原理

通过 HTTP Header (如 `X-SMG-Routing-Key`) 实现会话亲和 (Sticky Session)。同一 routing key 的请求始终路由到同一 Worker,实现类似"SessionAffinity"的效果。

> **名称更正**: 早期文档将此功能称为 "SessionAffinity",实际代码中的策略名为 `manual`。Manual 策略通过 routing key 映射实现会话亲和,同时支持手动指定目标 Worker。

#### 决策流程

```
请求到达
  ↓
[优先级 1] 检查 X-SMG-Target-Worker Header
  ├─ 存在 → 直接路由到指定 Worker
  ↓
[优先级 2] 检查 X-SMG-Routing-Key Header
  ├─ 存在 → 查询 routing key → Worker 映射
  │   ├─ 已映射 → 返回绑定的 Worker (会话亲和)
  │   └─ 未映射 → 按 assignment_mode 分配新 Worker,记录映射
  ↓
[优先级 3] 无 routing key → 按 assignment_mode 随机分配
```

#### 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `eviction_interval_secs` | 60 | TTL 清理周期 (秒) |
| `max_idle_secs` | 14400 (4小时) | 最大空闲时间,超时清理映射 |
| `assignment_mode` | random | 新 routing key 的分配模式: random / min_load / min_group |

#### 配置示例

```json
{
  "policy": {
    "type": "manual",
    "eviction_interval_secs": 60,
    "max_idle_secs": 14400,
    "assignment_mode": "random"
  }
}
```

#### 客户端使用示例

```bash
# 会话亲和: 同一 user_id 的请求始终路由到同一 Worker
curl -X POST http://gateway:3000/v1/chat/completions \
  -H "X-SMG-Routing-Key: user-123" \
  -d '{"messages": [{"role": "user", "content": "你好"}]}'

# 手动指定 Worker: 直接路由到特定 Worker
curl -X POST http://gateway:3000/v1/chat/completions \
  -H "X-SMG-Target-Worker: 0" \
  -d '{"messages": [{"role": "user", "content": "调试请求"}]}'
```

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| 多轮对话 | ★★★★★ |
| 流式交互 | ★★★★★ |
| 用户会话绑定 | ★★★★★ |
| 调试/测试 | ★★★★★ |

#### 优缺点

**优点**:
- 保持 KV Cache 连续 (同一会话路由到同一 Worker)
- 减少上下文迁移开销
- 支持手动指定 Worker (调试用)
- LRU 自动清理过期会话

**缺点**:
- 可能导致负载不均 (热门会话集中在一个 Worker)
- 需要客户端配合设置 routing key
- 映射表内存开销 (受 max_idle_secs 控制)

#### 调优建议

- **空闲超时**: 根据业务场景调整 `max_idle_secs` (对话场景建议 30-60 分钟)
- **分配模式**: 负载均衡场景用 `min_load`,简单场景用 `random`
- **故障转移**: Worker 下线时,映射自动失效,下次请求重新分配

### 4.2 ConsistentHashing (一致性哈希) ⚠️ 仅配置文件可用

> **名称更正**: 早期文档称此策略为 "ConsistentHash",实际代码中的名称为 `consistent_hashing`。该策略存在于 PolicyConfig 枚举中,但未暴露给 `--policy` CLI 参数,只能通过配置文件使用。

#### 基本原理

使用一致性哈希算法将请求映射到 Worker,减少 Worker 变化时的重新映射。与 Manual 策略类似,也通过 `X-SMG-Routing-Key` Header 实现会话亲和,但使用哈希环而非映射表。

#### 与 Manual 策略的区别

| 维度 | Manual | ConsistentHashing |
|------|--------|-------------------|
| **数据结构** | DashMap 映射表 | 哈希环 |
| **映射方式** | 显式记录 key→Worker | Hash(key) → 环查找 |
| **Worker 下线** | 映射失效,重新分配 | 仅 ~1/N 会话迁移 |
| **内存开销** | 中等 (映射表) | 极低 (环) |
| **CLI 可用** | ✅ | ❌ (仅配置文件) |

#### 适用场景

- **大规模分布式系统**
- **Worker 频繁上下线**
- **需要最小化会话迁移**
- **通过配置文件而非 CLI 启动**

## 五、异构 GPU 调度策略 ← 新增重点

### 5.1 RequestSizeBucket (请求长度分桶)

#### 基本原理

根据请求的输入长度将请求分到不同的桶,每个桶对应不同特性的 Worker。

#### 决策流程

```
请求到达
  ↓
提取输入长度 (prompt.length)
  ↓
if length < short_threshold:
    bucket = Short
elif length < medium_threshold:
    bucket = Medium
else:
    bucket = Long
  ↓
选择对应桶中负载最小的 Worker
```

#### 伪代码

```rust
struct RequestSizeBucket {
    short_threshold: usize,      // 短请求阈值 (默认 100)
    medium_threshold: usize,     // 中等请求阈值 (默认 500)
    track_load_per_bucket: bool, // 是否分桶跟踪负载
}

impl LoadBalancingPolicy for RequestSizeBucket {
    fn select_worker(&self, request: &Request) -> Option<WorkerId> {
        let input_len = request.prompt.len();
        
        let bucket = if input_len < self.short_threshold {
            Bucket::Short
        } else if input_len < self.medium_threshold {
            Bucket::Medium
        } else {
            Bucket::Long
        };
        
        // 在对应桶内选择负载最小的 Worker
        self.workers_in_bucket(bucket)
            .min_by_key(|w| w.active_requests)
            .map(|w| w.id)
    }
}
```

#### Worker 分配逻辑

```
按 Worker 的 priority/cost 比例分配桶:

score = priority / cost

高分数 (score > 5.0)  → Short 桶 (低延迟优化,适合高算力 GPU)
中分数 (1.0-5.0)      → Medium 桶 (均衡型)
低分数 (score < 1.0)  → Long 桶 (大显存优化,适合大显存 GPU)
```

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| Prompt 长度差异大 | ★★★★★ |
| Prefill 阶段调度 | ★★★★★ |
| H800+H20 混部 (Prefill 层) | ★★★★★ |

#### 优缺点

**优点**:
- 隔离不同长度请求的负载
- 避免长请求阻塞短请求
- 简单高效

**缺点**:
- 需要预先知道请求长度分布
- 阈值需要调优
- 不考虑输出长度

#### 调优建议

- **阈值调整**: 根据实际请求分布调整,使三个桶的请求数大致均匀
- **桶数量**: 可扩展为多桶 (如 5 桶),但增加复杂度
- **Worker 分配**: 高算力 GPU 分配给短请求桶 (追求低延迟),大显存 GPU 分配给长请求桶 (追求吞吐)

### 5.2 PerformanceAware (性能感知)

#### 基本原理

收集每个 Worker 的历史性能指标 (TTFT、TPOT、吞吐量),动态选择最优 Worker。

#### 决策流程

```
请求到达
  ↓
收集各 Worker 性能指标:
  - TTFT (Time To First Token)
  - TPOT (Time Per Output Token)
  - Throughput (吞吐量)
  ↓
归一化指标到 [0, 1]
  ↓
计算加权分数:
  score = w_ttft * norm_ttft + w_tpot * norm_tpot + w_throughput * norm_throughput
  ↓
考虑当前负载:
  final_score = score * load_factor
  load_factor = 1.0 / (1.0 + active_requests)
  ↓
选择分数最高的 Worker
```

#### 伪代码

```rust
struct PerformanceAware {
    weight_ttft: f64,             // TTFT 权重 (默认 0.3)
    weight_tpot: f64,             // TPOT 权重 (默认 0.3)
    weight_throughput: f64,       // 吞吐量权重 (默认 0.4)
    score_refresh_interval: u64,  // 分数刷新间隔 (默认 60s)
    consider_load: bool,          // 是否考虑负载
}

impl LoadBalancingPolicy for PerformanceAware {
    fn select_worker(&self, request: &Request) -> Option<WorkerId> {
        // 获取指标范围
        let min_ttft = self.workers.iter().map(|w| w.ttft).min()?;
        let max_ttft = self.workers.iter().map(|w| w.ttft).max()?;
        let min_tpot = self.workers.iter().map(|w| w.tpot).min()?;
        let max_tpot = self.workers.iter().map(|w| w.tpot).max()?;
        let min_tp = self.workers.iter().map(|w| w.throughput).min()?;
        let max_tp = self.workers.iter().map(|w| w.throughput).max()?;
        
        // 计算每个 Worker 的分数
        let best_worker = self.workers
            .iter()
            .map(|w| {
                let norm_ttft = 1.0 - (w.ttft - min_ttft) / (max_ttft - min_ttft);
                let norm_tpot = 1.0 - (w.tpot - min_tpot) / (max_tpot - min_tpot);
                let norm_tp = (w.throughput - min_tp) / (max_tp - min_tp);
                
                let score = self.weight_ttft * norm_ttft
                          + self.weight_tpot * norm_tpot
                          + self.weight_throughput * norm_tp;
                
                let load_factor = if self.consider_load {
                    1.0 / (1.0 + w.active_requests as f64)
                } else {
                    1.0
                };
                
                (w.id, score * load_factor)
            })
            .max_by(|a, b| a.1.partial_cmp(&b.1).unwrap());
        
        best_worker.map(|(id, _)| id)
    }
}
```

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| Decode 阶段调度 | ★★★★★ |
| H800+H20 混部 (Decode 层) | ★★★★★ |
| Worker 性能波动 | ★★★★★ |
| 需要实际性能数据 | ★★★★☆ |

#### 优缺点

**优点**:
- 基于实际性能,不是理论值
- 自动适应性能变化
- 多目标优化 (延迟 + 吞吐)

**缺点**:
- 需要积累性能数据 (冷启动问题)
- 指标收集有开销
- 权重需要调优

#### 调优建议

- **权重调整**:
  - 延迟敏感场景: `weight_ttft=0.5, weight_tpot=0.3, weight_throughput=0.2`
  - 吞吐敏感场景: `weight_ttft=0.2, weight_tpot=0.3, weight_throughput=0.5`
  - 均衡场景: `weight_ttft=0.3, weight_tpot=0.3, weight_throughput=0.4`

- **刷新间隔**: 
  - 性能稳定: 60-120 秒
  - 性能波动大: 10-30 秒

- **冷启动**: 初始使用 RoundRobin,积累数据后切换到 PerformanceAware

### 5.3 RequestClassification (请求分类) ← 重点新增

#### 基本原理

根据请求的输入长度和预期输出长度,将请求分类为计算密集型、内存密集型或均衡型,然后路由到最适合的 Worker。

#### 决策流程

```
请求到达
  ↓
提取:
  - input_length (prompt 长度)
  - expected_output_length (max_tokens)
  ↓
分类逻辑:
  if input < 100 and output > 100:
      type = ComputeIntensive  (计算密集型)
  elif input > 500 and output < 500:
      type = MemoryIntensive   (内存密集型)
  else:
      type = Balanced          (均衡型)
  ↓
路由:
  ComputeIntensive → 高算力 GPU (如 H800)
  MemoryIntensive  → 大显存 GPU (如 H20)
  Balanced         → 均衡 GPU
```

#### 伪代码

```rust
struct RequestClassification {
    short_input_threshold: usize,    // 短输入阈值 (默认 100)
    medium_input_threshold: usize,   // 中等输入阈值 (默认 500)
    small_output_threshold: usize,   // 小输出阈值 (默认 100)
    medium_output_threshold: usize,  // 中等输出阈值 (默认 500)
    auto_assign_workers: bool,       // 自动分配 Worker
}

enum RequestType {
    ComputeIntensive,  // 计算密集型
    MemoryIntensive,   // 内存密集型
    Balanced,          // 均衡型
}

impl LoadBalancingPolicy for RequestClassification {
    fn select_worker(&self, request: &Request) -> Option<WorkerId> {
        let input_len = request.prompt.len();
        let output_len = request.max_tokens.unwrap_or(256);
        
        // 分类请求
        let request_type = if input_len < self.short_input_threshold 
                           && output_len > self.small_output_threshold {
            RequestType::ComputeIntensive
        } else if input_len > self.medium_input_threshold 
                  && output_len < self.medium_output_threshold {
            RequestType::MemoryIntensive
        } else {
            RequestType::Balanced
        };
        
        // 选择对应类型的 Worker
        let workers = self.workers_by_type(request_type);
        workers
            .into_iter()
            .min_by_key(|w| w.active_requests)
            .map(|w| w.id)
    }
}

fn workers_by_type(&self, req_type: RequestType) -> Vec<&Worker> {
    if self.auto_assign_workers {
        // 自动分配: 根据 worker 的 priority/cost 分类
        self.workers.iter().filter(|w| {
            let score = w.priority as f64 / w.cost as f64;
            match req_type {
                RequestType::ComputeIntensive => score > 5.0,  // 高算力
                RequestType::MemoryIntensive => score < 1.0,   // 大显存
                RequestType::Balanced => (1.0..=5.0).contains(&score),
            }
        }).collect()
    } else {
        // 手动分配: 使用 worker 标签
        self.workers.iter().filter(|w| w.worker_type == req_type).collect()
    }
}
```

#### 分类逻辑详解

| 类型 | 输入长度 | 输出长度 | 典型场景 | 路由目标 |
|------|---------|---------|---------|---------|
| **计算密集型** | < 100 | > 100 | 创意写作、代码生成、翻译 | 高算力 GPU (H800) |
| **内存密集型** | > 500 | < 500 | 文档摘要、问答、分类 | 大显存 GPU (H20) |
| **均衡型** | 100-500 | 100-500 | 对话、中等文本生成 | 均衡 GPU |

#### 测试结果

**测试环境**: WSL2 + RTX 4070 Ti SUPER 16GB  
**测试场景**: 三种类型请求各 1 个

| 请求类型 | 输入长度 | 输出 Tokens | 响应时间 | 状态 |
|---------|---------|-------------|---------|------|
| Short (计算密集) | 5 字符 | 50 | 424ms | ✅ 200 |
| Medium (均衡) | 200 字符 | 200 | 1354ms | ✅ 200 |
| Long (内存密集) | 1000 字符 | 50 | 3020ms | ✅ 200 |

**成功率**: 100% (3/3)

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| 请求类型多样 | ★★★★★ |
| 异构 GPU 环境 | ★★★★★ |
| H800+H20 混部 | ★★★★★ |
| 需要智能路由 | ★★★★★ |

#### 优缺点

**优点**:
- 智能识别请求特征
- 充分利用异构 GPU 优势
- 自动化程度高

**缺点**:
- 需要预估输出长度 (可能不准确)
- 分类边界需要调优
- 依赖 Worker 元数据 (priority/cost)

#### 调优建议

- **阈值调整**: 根据实际请求分布调整输入/输出阈值
- **Worker 分类**: 如果 Worker 特性不明显,关闭 `auto_assign_workers`,手动指定标签
- **输出长度预估**: 
  - 使用历史数据训练预估模型
  - 或提供默认值 (如 256 tokens)

## 六、PD 分离架构下的策略组合 ⚠️ 无独立 PDScheduler

> **⚠️ 重要**: `PDScheduler` 策略在当前代码库中**不存在**作为独立策略。PD (Prefill-Decode) 分离架构下的调度是通过为 Prefill 层和 Decode 层**分别配置不同策略**实现的,而非一个统一的 "PDScheduler" 策略。
>
> Gateway 在 PD 模式下支持分别设置 `--prefill-policy` 和 `--decode-policy`,其效果等同于文档描述的 "PDScheduler"。

### 6.1 PD 分离架构策略配置

#### 基本原理

在 Prefill-Decode 分离架构中,为两个阶段分别配置不同的调度策略,最大化各阶段性能。

#### 架构

```
┌─────────────────────────────────────────────────────┐
│                  Gateway (调度层)                      │
│                                                     │
│  Prefill 层调度策略: RequestSizeBucket                 │
│  ├─ Prefill-1 (H800) - 处理短请求                     │
│  ├─ Prefill-2 (H800) - 处理长请求                     │
│  └─ Bootstrap 传输 KV Cache → Decode 层               │
│                                                     │
│  Decode 层调度策略: PerformanceAware                   │
│  ├─ Decode-1 (H20) - 高吞吐解码                       │
│  ├─ Decode-2 (H20) - 高吞吐解码                       │
│  └─ Decode-3 (H20) - 高吞吐解码                       │
└─────────────────────────────────────────────────────┘
```

#### 决策流程

```
请求到达
  ↓
┌─ Prefill 阶段 ──────────────────────────────┐
│  1. 根据请求长度选择 Prefill Worker           │
│  2. 执行 Prefill,生成 KV Cache               │
│  3. 通过 Bootstrap 端口传输 KV Cache 到 Decode │
└─────────────────────────────────────────────┘
  ↓
┌─ Decode 阶段 ───────────────────────────────┐
│  1. 根据性能指标选择 Decode Worker            │
│  2. 接收 KV Cache                            │
│  3. 执行 Decode,逐 token 生成输出             │
└─────────────────────────────────────────────┘
  ↓
返回结果
```

#### 配置示例

```json
{
  "mode": {
    "type": "prefill_decode",
    "prefill_urls": [
      ["http://prefill-1:30000", 9000],
      ["http://prefill-2:30001", 9001]
    ],
    "decode_urls": [
      "http://decode-1:30010",
      "http://decode-2:30011",
      "http://decode-3:30012"
    ],
    "prefill_policy": {
      "type": "request_size_bucket",
      "short_threshold": 100,
      "medium_threshold": 500
    },
    "decode_policy": {
      "type": "performance_aware",
      "weight_ttft": 0.2,
      "weight_tpot": 0.5,
      "weight_throughput": 0.3
    }
  }
}
```

#### 适用场景

| 场景 | 适配度 |
|------|-------|
| H800+H20 异构混部 | ★★★★★ |
| Prefill 和 Decode 瓶颈不同 | ★★★★★ |
| 需要最大化吞吐量 | ★★★★★ |

#### 优缺点

**优点**:
- 充分发挥各 GPU 优势
- Prefill 和 Decode 独立优化
- 整体性能最大化

**缺点**:
- 架构复杂
- 需要 Bootstrap 网络传输
- 调优难度大

## 九、调优指南

### 9.1 关键参数调优路径

#### RequestSizeBucket

```
调优步骤:
1. 收集请求长度分布 (P10, P50, P80, P90)
2. 设置 short_threshold = P30
3. 设置 medium_threshold = P70
4. 观察三个桶的负载是否均衡
5. 如果不均衡,调整阈值使分布均匀
```

**示例**:
```
实际分布: P10=50, P30=100, P50=200, P70=500, P90=1000
推荐配置:
  short_threshold = 100   (P30)
  medium_threshold = 500  (P70)
预期分布:
  Short 桶: 30% 请求
  Medium 桶: 40% 请求
  Long 桶: 30% 请求
```

#### PerformanceAware

```
调优步骤:
1. 确定业务目标 (延迟优先 or 吞吐优先)
2. 设置初始权重:
   - 延迟优先: TTFT 0.5, TPOT 0.3, Throughput 0.2
   - 吞吐优先: TTFT 0.2, TPOT 0.3, Throughput 0.5
   - 均衡: TTFT 0.3, TPOT 0.3, Throughput 0.4
3. 运行压测,收集指标
4. 根据实际效果微调权重
5. 设置合理的刷新间隔 (性能稳定则长,波动大则短)
```

**示例**:
```
场景: H800+H20 混部,Decode 层
目标: 最大化吞吐,兼顾延迟
配置:
  weight_ttft = 0.2
  weight_tpot = 0.5    # TPOT 最重要 (Decode 阶段)
  weight_throughput = 0.3
  score_refresh_interval_secs = 60
  consider_load = true
```

#### RequestClassification

```
调优步骤:
1. 分析历史请求,统计输入/输出长度分布
2. 设置分类阈值:
   - short_input_threshold: 短输入分界点
   - medium_input_threshold: 中等输入分界点
   - small_output_threshold: 小输出分界点
   - medium_output_threshold: 中等输出分界点
3. 观察分类准确率
4. 根据 Worker 实际负载调整分类边界
5. 如果 Worker 特性明显,开启 auto_assign_workers;否则手动指定
```

**示例**:
```
场景: 创意写作平台
请求特征:
  - 提示词短 (10-50 字符)
  - 输出长 (500-2000 tokens)
配置:
  short_input_threshold = 100
  medium_input_threshold = 500
  small_output_threshold = 100
  medium_output_threshold = 500
  auto_assign_workers = true  # H800 自动识别为计算密集型
```

### 9.2 常见误区

| 误区 | 正确做法 |
|------|---------|
| **使用 RoundRobin 处理异构环境** | 异构环境必须用异构调度策略 |
| **PerformanceAware 冷启动就用** | 冷启动先用 RoundRobin,积累数据后再切换 |
| **RequestSizeBucket 阈值随意设置** | 必须基于实际请求分布设置 |
| **忽略输出长度只看输入长度** | RequestClassification 同时考虑输入和输出 |
| **SessionAffinity 不设超时** | 必须设置超时清理,否则内存泄漏 |
| **CacheAware 不考虑缓存淘汰** | 必须配置 LRU/LFU,否则显存爆满 |
| **PD 分离不使用 Bootstrap** | PD 分离必须配置 Bootstrap 端口传输 KV Cache |
| **策略参数一次调优永不改** | 定期复盘,根据业务变化调整 |

### 9.3 性能监控指标

| 指标 | 说明 | 告警阈值 | 优化策略 |
|------|------|---------|---------|
| **TTFT** | 首 Token 延迟 | > 2s | 增加 Prefill 节点,优化路由策略 |
| **TPOT** | 每 Token 延迟 | > 100ms | 增加 Decode 节点,检查 GPU 利用率 |
| **Throughput** | 吞吐量 (tokens/s) | < 预期 50% | 检查负载是否均衡,调整策略参数 |
| **Cache Hit Rate** | 缓存命中率 | < 20% | 检查缓存策略,调整淘汰算法 |
| **Session Migration Rate** | 会话迁移率 | > 5% | 检查 SessionAffinity 配置 |
| **Load Imbalance** | 负载不均衡度 | > 30% | 调整策略权重,检查 Worker 健康 |

## 十、常见问题 FAQ

### Q1: 如何选择 RoundRobin 和 PowerOfTwo?

**A**: 
- **小规模集群 (N < 10)** 且请求长度均匀 → RoundRobin
- **中大规模集群 (N > 10)** 或请求长度有差异 → PowerOfTwo
- PowerOfTwo 的负载均衡效果接近理想的 LeastLoad,但性能开销只有 O(1)

> 注: LeastLoad 策略不存在,其功能由 PowerOfTwo 替代。

### Q2: PerformanceAware 冷启动问题怎么解决?

**A**: 
有三种方案:
1. **渐进式**: 前 100 个请求用 RoundRobin,积累数据后切换到 PerformanceAware
2. **预估数据**: 使用 Worker 的理论性能指标 (TFLOPS、带宽) 作为初始值
3. **快速压测**: 启动时自动发送一批测试请求,快速收集指标

推荐方案 1,实现简单且安全。

### Q3: RequestSizeBucket 和 RequestClassification 有什么区别?

**A**:

| 维度 | RequestSizeBucket | RequestClassification |
|------|-------------------|----------------------|
| **分类依据** | 只看输入长度 | 同时看输入和输出长度 |
| **分类数量** | 3 类 (短/中/长) | 3 类 (计算密集/内存密集/均衡) |
| **适用场景** | Prefill 阶段 | 完整请求生命周期 |
| **复杂度** | 简单 | 中等 |
| **准确度** | 中等 (忽略输出长度) | 高 (考虑完整特征) |

**选择建议**:
- 如果只关心 Prefill 阶段 → RequestSizeBucket
- 如果需要端到端优化 → RequestClassification
- PD 分离架构 → Prefill 用 RequestSizeBucket,Decode 用 PerformanceAware

### Q4: Manual 策略的会话亲和会导致负载不均怎么办?

**A**:

三种解决方案:
1. **负载失衡阈值**: 当某 Worker 负载超过平均值 150% 时,新会话路由到其他 Worker
2. **会话超时**: 30-60 分钟无活动的会话自动解绑
3. **动态重平衡**: 定期 (如每小时) 重新分配会话

```json
{
  "policy": {
    "type": "manual",
    "eviction_interval_secs": 60,
    "max_idle_secs": 1800,
    "assignment_mode": "min_load"
  }
}
```

> 注: 早期文档中的 "SessionAffinity" 策略不存在,请使用 `manual` 策略并通过 `X-SMG-Routing-Key` Header 实现会话亲和。

### Q5: PD 分离架构中,Bootstrap 传输会成为瓶颈吗?

**A**:

**可能,但可以优化**:

1. **网络带宽**: 确保 Prefill 和 Decode 节点之间有 10Gbps+ 网络
2. **传输协议**: 使用 RDMA 或 TCP 零拷贝技术
3. **KV Cache 压缩**: 减少传输数据量 (如量化到 INT8)
4. **异步传输**: Prefill 完成后立即传输,不阻塞下一个请求

**监控指标**:
- Bootstrap 传输延迟 (应 < 10ms)
- 网络带宽利用率 (应 < 80%)
- KV Cache 大小 (提前预估)

**示例**:
```
Qwen3 30B 模型:
  - 输入 1000 tokens → KV Cache ~ 50MB
  - 10Gbps 网络 → 传输时间 ~ 40ms
  - 可接受范围: < 100ms
```

### D.1 Worker 负载维护机制

#### 通俗理解

想象你有 **5 个服务员（Worker）**，Gateway 是 **领班**，需要知道每个服务员手里有几桌客人。

#### 实际机制

**1. 本地计数器（主动跟踪）**

```rust
// 每个 Worker 自己维护一个计数器
struct Worker {
    load_counter: AtomicUsize,  // 当前活跃请求数
}

// 请求来了 +1
fn increment_load(&self) {
    self.load_counter.fetch_add(1, Ordering::Relaxed);
}

// 请求走了 -1
fn decrement_load(&self) {
    self.load_counter.fetch_sub(1, Ordering::Relaxed);
}
```

**流程：**
- 请求到达 → Gateway 分配给 Worker → `load_counter + 1`
- 请求处理完 → Worker 返回结果 → `load_counter - 1`
- **优点**：实时准确，零网络开销

**2. 后端负载查询（被动获取）**

```rust
// Gateway 定期请求 Worker 的 /get_load 接口
async fn parse_load_response(client, url) -> isize {
    let load_url = format!("{}/get_load", url);
    // 获取 Worker 自己报告的负载值
}
```

**流程：**
- `LoadMonitor` 定时器（比如每 5 秒）→ 请求所有 Worker 的 `/get_load` 接口
- 收集到的负载值 → 更新给 `PowerOfTwo` 等策略使用
- **优点**：能获取 Worker 内部更精确的负载（如 GPU 利用率、队列长度）

**3. 健康检查（辅助判断）**

```rust
// 定期检查 Worker 是否健康
fn check_health(&self) -> bool {
    // 连续失败 N 次 → 标记不健康 → 不再分配请求
}
```

#### 负载数据流向

```
┌─────────────┐
│   请求到达   │
└──────┬──────┘
       │
       ▼
┌─────────────────────────────────┐
│  Worker.increment_load()        │
│  load_counter.fetch_add(1)      │  ← 实时计数器
└─────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│  策略读取负载                    │
│  - LeastLoad: 选最小负载         │
│  - PowerOfTwo: 比较 2 个 Worker  │
│  - CacheAware: 检查负载均衡度    │
└─────────────────────────────────┘
       │
       ▼
┌─────────────┐
│  请求完成    │
└──────┬──────┘
       │
       ▼
┌─────────────────────────────────┐
│  Worker.decrement_load()        │
│  load_counter.fetch_sub(1)      │  ← 释放负载
└─────────────────────────────────┘
```

### D.2 Cache 状态感知机制

#### 通俗理解

Gateway **不直接查 GPU 的 KV Cache**，而是通过 **历史请求记录** 来"猜"哪个 Worker 可能缓存了相关数据。

#### 实际机制：近似 Radix Tree（前缀树）

```rust
struct CacheAwarePolicy {
    // 每个模型维护一棵树
    trees: DashMap<String, Arc<Tree>>,  // model_id → Tree
}

// Tree 的结构（简化）
struct Tree {
    root: Node,  // 前缀树根节点
}

struct Node {
    text: String,           // 存储的文本片段
    tenant: String,         // Worker URL（哪个 Worker 处理过）
    children: DashMap<char, Node>,  // 子节点
    last_access: Instant,   // 最后访问时间（用于 LRU 淘汰）
}
```

**工作流程：**

```
请求: "今天天气怎么样？"

1. 在 Tree 中查找最长前缀匹配
   └─ 找到 "今天天" → Worker-1 之前处理过

2. 计算匹配率
   └─ 匹配 3 字符 / 总共 7 字符 = 43%

3. 决策：
   - 如果匹配率 > 阈值（如 50%）→ 路由给 Worker-1（缓存命中）
   - 如果匹配率 < 阈值 → 路由给负载最小的 Worker
   - 如果负载不均衡 → 直接选负载最小的，忽略缓存

4. 请求完成后，把 "今天天气怎么样？" 插入 Tree
   └─ 记录是哪个 Worker 处理的
```

**关键点：**
- **不查 GPU Cache**：Gateway 不访问 GPU 显存，只维护自己的内存 Tree
- **按模型隔离**：不同模型的 Tree 分开维护
- **LRU 淘汰**：定期删除最少使用的叶子节点，防止内存无限增长
- **Mesh 同步**（可选）：多 Gateway 实例时，通过 gossip 协议同步 Tree 状态

#### Cache Tree 维护流程

```
┌──────────────────────────────────────────────┐
│            Cache-Aware 策略启动               │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│  初始化：为每个模型创建空 Tree                 │
│  trees: {                                     │
│    "qwen-7b": Tree::new(),                   │
│    "llama-13b": Tree::new()                  │
│  }                                            │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│  请求到达："今天天气怎么样？"                  │
│  1. tree.prefix_match("今天天气怎么样？")     │
│  2. 返回: tenant=Worker-1, match_rate=43%    │
│  3. 根据 match_rate 决策路由                  │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│  请求完成后：                                  │
│  tree.insert("今天天气怎么样？", Worker-1)    │
│  → 更新 Tree，记录这个前缀在 Worker-1         │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│  后台定时任务（每 30 秒）：                    │
│  tree.evict_tenant_by_size(max_size=10000)   │
│  → 删除最少使用的叶子节点                      │
└──────────────────────────────────────────────┘
```

### D.3 KV Cache 中的 Block 概念 (CacheAware 相关)

> **⚠️ 提示**: 早期文档将此节标题为 "CacheLocality 中的 Block 概念",但 CacheLocality 策略不存在。Block 概念属于 CacheAware 策略的底层知识。

#### 通俗理解

在 KV Cache 中，**Block 是显存的基本单位**（类似内存页）。

#### 详细解释

**1. Block 是什么？**

```
GPU 显存不是按 token 分配的，而是按 Block 分配：
- 1 个 Block = 固定数量的 token（如 16 个 token）
- 类似操作系统的"页"（Page），一页 = 4KB

例如：
Token 序列: [t1, t2, t3, ..., t100]
Block 0:    [t1, t2, ..., t16]      ← 第 1 个 Block
Block 1:    [t17, t18, ..., t32]    ← 第 2 个 Block
Block 2:    [t33, t34, ..., t48]    ← 第 3 个 Block
...
```

**2. "相邻 Block"是什么？**

```
如果 Worker-1 的缓存中有:
Block 5: "今天天气不错，我们"
Block 6: "去公园散步吧？"

这两个 Block 在原文中是连续的 → 称为"相邻 Block"

CacheLocality 的核心思想：
- 如果请求的前缀匹配了 Block 5
- 那么很可能也需要 Block 6
- 优先路由到有连续 Block 的 Worker
```

**3. 为什么相邻 Block 重要？**

```
场景：多轮对话
Round 1: "帮我写一首诗"          → Block 0-3
Round 2: "帮我写一首诗，关于春天" → Block 0-5

如果 Round 1 在 Worker-1，Round 2 也路由到 Worker-1：
✅ 直接复用 Block 0-3，只需生成 Block 4-5
❌ 如果路由到 Worker-2，需要重新计算 Block 0-3

相邻 Block 的局部性越好 → 缓存命中率越高 → 延迟越低
```

**注意**：从代码来看，SGLang Gateway 当前主要使用 **Prefix Match（前缀匹配）**（CacheAware 策略的 Radix Tree），而没有显式实现 Block 级别的局部性优化。CacheLocality 作为独立策略不存在，其概念已由 CacheAware 涵盖。

### D.4 PrefixHash 前缀路由机制

> **名称更正**: 早期文档标题为 "PrefixCache 前缀缓存表维护",实际代码中的策略名为 `prefix_hash`。PrefixHash 使用一致性哈希环实现前缀路由,而非 Radix Tree。本节内容描述的是 CacheAware 的 Radix Tree 机制,保留供参考。

#### 通俗理解

就像**字典的索引页**，按首字母快速查找。

#### 实际机制：Multi-Tenant Radix Tree（多租户基数树）

**1. 数据结构**

```rust
// 以字符为单位的压缩前缀树
Tree {
    root ── "hel" ── "lo world" (Worker-1)
          │
          ├─ "lo" (Worker-2)
          │
          └─ "help" ── " me" (Worker-1)
}

// 压缩存储：
// "hello world" 和 "help me" 共享前缀 "hel"
// 不用每个字符一个节点，而是合并连续字符
```

**2. 插入流程**

```rust
// 请求: "今天天气怎么样？" 路由给了 Worker-1
tree.insert("今天天气怎么样？", "http://worker-1:8000");

Tree 结构变化：
root

```

### D.5 一致性 Hash 策略 (ConsistentHashing) 完整流程

> **名称更正**: 代码中的实际名称为 `consistent_hashing`,该策略存在于 PolicyConfig 枚举但未暴露给 CLI,只能通过配置文件使用。

#### 解决的问题

**问题 1：会话亲和性**

```
用户 A 的多轮对话：
Round 1: "你好" → Worker-1
Round 2: "继续说" → ??? 

如果用轮询：可能路由到 Worker-2
❌ Worker-2 没有 Worker-1 的缓存，需要重新计算

用一致性 Hash：
Round 2: "继续说" → Worker-1（因为 hash("用户A") → Worker-1）
✅ 保持缓存连续性
```

**问题 2：Worker 上下线时的最小化迁移**

```
4 个 Worker 时：100 个用户会话均匀分布
每个 Worker 处理 25 个会话

Worker-2 宕机了：
❌ 如果用普通 Hash：所有用户重新 Hash，75% 的会话迁移
✅ 如果用一致性 Hash：只有 Worker-2 上的 25 个会话迁移（1/4）
```

#### 实际流程

**1. 构建 Hash Ring（环）**

```rust
// 假设有 3 个 Worker
workers = ["W1", "W2", "W3"]

// 每个 Worker 映射到环上的一个位置（0-2^64）
ring = HashRing::new(&workers);

位置分布：
         0 ─────────────────── 2^64
          │
   W2 (1.2×10^18)
          │
          │     W1 (3.4×10^18)
          │          │
          │          │   W3 (5.6×10^18)
          │          │        │
```

**2. 请求路由**

```rust
// 用户请求携带 routing_key（如 user_id）
routing_key = "user-123"

// 1. Hash 计算
hash_value = hash("user-123") = 2.8×10^18

// 2. 在环上顺时针查找第一个 Worker
         0 ─────────────────── 2^64
          │
   W2 (1.2×10^18)
          │
          │  ← hash(user-123) = 2.8×10^18
          │     │
          │     └─ 顺时针找 →  W1 (3.4×10^18) ← 选中！

// 3. 返回 Worker-1
```

**3. Worker 宕机时的处理**

```rust
// W1 宕机了，标记为不健康
workers[1].set_healthy(false);

// 重新查找 "user-123"
hash_value = 2.8×10^18

// 顺时针查找，跳过不健康的 W1
         0 ─────────────────── 2^64
          │
   W2 (1.2×10^18)
          │
          │  ← 2.8×10^18
          │     │
          │     └─ W1 (宕机，跳过)
          │            │
          │            └─ W3 (5.6×10^18) ← 新的目标

// 只有原来在 W1 上的会话需要迁移，其他用户不受影响
```

**4. 代码实现的关键点**

```rust
// 优先级 1: 直接指定 Worker（调试用）
if let Some(idx) = extract_target_worker(headers) {
    return workers[idx];  // X-SMG-Target-Worker: 1
}

// 优先级 2: 显式 Routing Key
if let Some(key) = extract_routing_key(headers) {
    return ring.find_healthy_url(key);  // X-SMG-Routing-Key: user-123
}

// 优先级 3: 隐式 Routing Key（从 HTTP Header 推断）
let implicit_key = headers.get("authorization")  // 用户 Token
                .or(headers.get("x-forwarded-for"))  // IP 地址
                .or(headers.get("cookie"));  // Cookie

// 优先级 4: 随机选择（完全没有标识）
return random_healthy_worker();
```

#### 一致性 Hash 路由决策树

```
请求到达
  │
  ▼
有 X-SMG-Target-Worker Header？
  ├─ 是 → 检查索引是否合法 & Worker 是否健康
  │        ├─ 是 → 返回该 Worker (TargetWorkerHit)
  │        └─ 否 → 返回错误 (TargetWorkerMiss)
  │
  └─ 否 → 有 X-SMG-Routing-Key Header？
           ├─ 是 → Hash Ring 查找
           │        └─ 返回命中的 Worker (RoutingKeyHit)
           │
           └─ 否 → 有隐式 Routing Key？
                    (Authorization / X-Forwarded-For / Cookie)
                    ├─ 是 → Hash Ring 查找
                    │        └─ 返回命中的 Worker (RoutingKeyHit)
                    │
                    └─ 否 → 随机选择健康 Worker
                             (RandomFallback)
```

#### 最小化重分布验证

```rust
// 测试：4 个 Worker，100 个用户会话
// Worker-1 宕机后，只有 ~25 个会话迁移（而不是 75 个）

let keys_on_failed_worker = 25;  // 原来在 Worker-1 的会话数
let moved_count = 27;            // 实际迁移的会话数（含少量 variance）

// 一致性 Hash 保证：moved_count ≈ keys_on_failed_worker
// 普通 Hash 会导致：moved_count ≈ 75（所有会话重新分配）
```

### D.6 核心机制对比总结

| 机制 | 维护方式 | 数据来源 | 实时性 | 开销 | 适用策略 |
|------|---------|---------|-------|------|----------|
| **Worker 负载** | 本地计数器 + 后端查询 | Gateway 计数 + Worker 报告 | 实时（本地）/ 周期（查询） | 极低 | PowerOfTwo, CacheAware |
| **Cache 状态** | 近似 Radix Tree | 历史请求记录（文本） | 准实时（请求后更新） | 中等（内存+CPU） | CacheAware |
| **前缀哈希** | 一致性哈希环 | 前缀 token hash | 实时 | 极低 (O(log n)) | PrefixHash |
| **Routing Key 映射** | DashMap | X-SMG-Routing-Key Header | 实时 | 极低 | Manual |
| **一致性 Hash** | Hash Ring | Worker URL + Routing Key | 实时 | 极低（O(log n)） | ConsistentHashing (仅配置) |

#### 内存占用估算

```
假设场景：
- 10 个 Worker
- 每个 Worker 处理过 1000 个不同请求
- 平均请求长度 100 字符

Tree 内存占用：
- 节点数：约 5000 个（压缩后）
- 每节点：~200 字节
- 总内存：5000 × 200 = 1MB

Hash Ring 内存占用：
- 每 Worker 1 个虚拟节点
- 总内存：10 × 64 字节 = 640 字节

负载计数器：
- 每 Worker 1 个 AtomicUsize
- 总内存：10 × 8 字节 = 80 字节
```

#### 性能开销对比

```
策略决策耗时（单次路由）：
- RoundRobin:        ~10 ns    （原子操作 +1）
- PowerOfTwo:        ~50 ns    （比较 2 个 Worker）
- CacheAware:        ~500 ns   （Tree 前缀匹配）
- PrefixHash:        ~100 ns   （Hash + 环查找）
- Manual:            ~50 ns    （DashMap 查询）
- ConsistentHashing: ~100 ns   （Hash + 二分查找）
- PerformanceAware:  ~200 ns   （分数计算，不含数据收集）
```
