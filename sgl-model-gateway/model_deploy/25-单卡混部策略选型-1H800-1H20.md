## 一、场景分析与架构选择

### 1.1 硬件约束

| GPU 型号 | 显存 | 算力 (FP16) | 带宽 | 数量 | 特点 |
|---------|------|------------|------|------|------|
| **H800** | 80GB HBM2e | 31 TFLOPS | 2.0 TB/s | **1 张** | 高算力,适合 Prefill 和计算密集型请求 |
| **H20** | 96GB HBM3 | 10 TFLOPS | 4.0 TB/s | **1 张** | 大显存,高带宽,适合 Decode 和长上下文 |

**关键差异**:
- **H800**: 算力是 H20 的 **3 倍**,但显存少 20%
- **H20**: 显存大 20%,带宽是 H800 的 **2 倍**,但算力只有 1/3

### 1.2 资源约束分析

**核心问题**: 只有 1 张 H800 + 1 张 H20,无法做多 Worker 调度!

| 架构模式 | Prefill Worker 数 | Decode Worker 数 | 可调度空间 |
|---------|------------------|-----------------|-----------|
| PD 分离 (2P+2D) | 2 | 2 | ✅ 有调度空间 |
| PD 分离 (1P+1D) | 1 | 1 | ❌ **无调度空间** |
| PD 不分离 | 2 (各自独立) | 2 (各自独立) | ✅ **有路由空间** |

**结论**: 
- PD 分离架构需要 **至少 2 个 Prefill Worker 或 2 个 Decode Worker** 才有调度意义
- 1P+1D 场景下,路由策略是 **无意义的** (只有 1 个目标可选)
- **必须切换到 PD 不分离架构**,让 H800 和 H20 各自运行完整模型,根据请求特征做路由决策

### 1.3 架构选择对比

| 维度 | PD 分离 (1P+1D) | PD 不分离 (2 独立 Worker) |
|------|----------------|-------------------------|
| **架构复杂度** | 高 (需 Bootstrap 传输) | 低 (独立部署) |
| **可调度空间** | ❌ 无 (只有 1 个目标) | ✅ 有 (H800 vs H20) |
| **硬件利用率** | 高 (专用化) | 中 (混合负载) |
| **网络要求** | 需要高速 RDMA | 无特殊要求 |
| **故障隔离** | 差 (级联故障) | 好 (独立故障) |
| **适用场景** | 大规模集群 | **小规模 / 单卡混部** |

**最终选择**: **PD 不分离架构** (2 独立 Worker)

## 二、为什么 PD 分离不再适用

### 2.1 PD 分离的核心前提

PD 分离架构有效的 **3 个前提条件**:

```
前提 1: 每层至少有 2 个 Worker
   ↓
   只有这样才能做负载均衡决策

前提 2: Prefill 和 Decode 的资源需求差异明显
   ↓
   Prefill: 算力密集 (TFLOPS)
   Decode:  带宽密集 (Memory Bandwidth)

前提 3: KV Cache 传输成本 < 分离带来的性能收益
   ↓
   需要高速网络 (RDMA / 10Gbps+)
```

### 2.2 1P+1D 场景的问题

**问题 1: 无调度空间**

```
请求到达 Gateway
  ↓
Prefill 层只有 1 个 Worker (P1)
  ↓
只能路由到 P1 (无选择)
  ↓
Decode 层只有 1 个 Worker (D1)
  ↓
只能路由到 D1 (无选择)
```

**结果**: 路由策略完全失效,退化为固定路由!

**问题 2: 额外复杂度**

- 需要配置 Bootstrap 端口传输 KV Cache
- 需要高速网络保证传输效率
- 故障排查复杂度增加

**问题 3: 单点故障**

```
P1 故障 → 所有请求失败 (无备用 Prefill)
D1 故障 → 所有请求失败 (无备用 Decode)
```

### 2.3 结论

**1P+1D 场景下,PD 分离架构的收益为 0,但复杂度增加 100%。**

**必须采用 PD 不分离架构!**

## 三、新技术路线:PD 不分离 + 智能路由

### 3.1 架构设计

```
                    ┌─────────────────────────────┐
                    │   Gateway (智能路由)          │
                    │   策略: RequestClassification │
                    │        or PerformanceAware   │
                    └──────────┬──────────────────┘
                               │
                    ┌──────────┴──────────┐
                    │                     │
              计算密集型请求         内存密集型请求
              (短输入+长输出)       (长输入+短输出)
                    │                     │
              ┌─────┴─────┐         ┌─────┴─────┐
              │  H800     │         │  H20      │
              │  :30000   │         │  :30010   │
              │  BF16     │         │  BF16     │
              │  完整模型  │         │  完整模型  │
              └───────────┘         └───────────┘
              Prefill + Decode     Prefill + Decode
              (混合负载)            (混合负载)
```

### 3.2 工作原理

```
1. 请求到达 Gateway
   ↓
2. 分析请求特征:
   - input_length (prompt tokens)
   - expected_output_length (max_tokens)
   - priority (可选)
   ↓
3. 分类请求类型:
   - 计算密集型: 短输入 + 长输出 → H800 (高算力)
   - 内存密集型: 长输入 + 短输出 → H20 (大显存 + 高带宽)
   - 均衡型: 中等输入 + 中等输出 → 负载较低的节点
   ↓
4. 路由到对应 GPU
   ↓
5. GPU 独立执行 Prefill + Decode (无 KV Cache 传输)
   ↓
6. 返回结果
```

### 3.3 架构优势

| 优势 | 说明 |
|------|------|
| **有调度空间** | H800 和 H20 是两个独立目标,可以做智能路由 |
| **部署简单** | 无需 Bootstrap 传输,无 RDMA 要求 |
| **故障隔离** | H800 故障时,H20 仍可服务 (反之亦然) |
| **灵活量化** | H800 可运行 FP16,H20 可运行 INT8 (按需选择) |
| **易于扩展** | 未来增加 GPU 时,直接加入路由池 |

### 3.4 架构劣势

| 劣势 | 影响 | 缓解措施 |
|------|------|---------|
| **Prefill 和 Decode 竞争资源** | 单卡同时做两种计算,可能互相干扰 | Chunked Prefill + 调度优化 |
| **H20 算力弱** | Decode 阶段 TPOT 可能较高 | 路由时避免计算密集型请求到 H20 |
| **需要预估输出长度** | RequestClassification 依赖 max_tokens | 提供默认值或训练预估模型 |

## 四、关键技术:Chunked Prefill

### 4.1 什么是 Chunked Prefill?

**问题背景**: 在 PD 不分离架构中,单卡同时承担 Prefill 和 Decode,存在资源竞争:

- **长 Prefill 阻塞 Decode**: 处理 10K tokens 的 prompt 需要数百毫秒,期间 Decode 请求等待
- **显存峰值高**: Prefill 阶段需要同时存储所有输入 token 的 KV Cache
- **GPU 利用率波动**: Prefill 时算力打满,Decode 时带宽打满,交替执行导致利用率不稳

**Chunked Prefill 解决方案**:
- 将长 prompt **切分为多个 chunk** (如每个 chunk 4096 tokens)
- 每个 chunk 在独立的调度周期中处理
- **Prefill 和 Decode 交错执行**,避免长时间阻塞

### 4.2 工作机制

```
传统 Prefill (无 Chunked):
┌──────────────────────────────────────────────┐
│ Prefill: 10000 tokens (耗时 500ms)            │
│ ┌──────────────────────────────────────────┐ │
│ │ 全部 token 一次性送入 GPU                  │ │
│ │ 期间 Decode 请求全部阻塞!                  │ │
│ └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
        ↓
┌──────────────────────────────────────────────┐
│ Decode: 开始生成 (TTFT = 500ms, 延迟高!)     │
└──────────────────────────────────────────────┘

Chunked Prefill:
┌──────────────────────────────────────────────┐
│ Chunk 1: 4096 tokens (200ms)                  │
│ ┌──────────────────────────────────────────┐ │
│ │ 处理第一批 token                           │ │
│ └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
        ↓ (交错执行)
┌──────────────────────────────────────────────┐
│ Decode Batch: 处理 5 个 Decode 请求 (50ms)    │
│ ┌──────────────────────────────────────────┐ │
│ │ 利用 Prefill 间隙,生成少量 token           │ │
│ └──────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
        ↓ (继续 Prefill)
┌──────────────────────────────────────────────┐
│ Chunk 2: 4096 tokens (200ms)                  │
└──────────────────────────────────────────────┘
        ↓
┌──────────────────────────────────────────────┐
│ Decode Batch: 处理 Decode 请求 (50ms)         │
└──────────────────────────────────────────────┘
        ↓
┌──────────────────────────────────────────────┐
│ Chunk 3: 1808 tokens (90ms)                   │
└──────────────────────────────────────────────┘
        ↓
┌──────────────────────────────────────────────┐
│ Decode: 开始生成 (TTFT = 200ms, 延迟降低!)   │
└──────────────────────────────────────────────┘
```

### 4.3 SGLang 配置

**启用方式**: SGLang **默认启用** Chunked Prefill,无需额外参数!

**关键参数**:

| 参数 | 说明 | 默认值 | H800 推荐 | H20 推荐 |
|------|------|--------|----------|---------|
| `--chunked-prefill-size` | 每个 chunk 的最大 token 数 | 8192 | **4096** | **2048** |
| `--max-prefill-tokens` | 单次 prefill 批次最大 token 数 | 16384 | **16384** | **8192** |
| `--enable-mixed-chunk` | 允许 prefill/decode 混合 | 默认开启 | **开启** | **开启** |

**H800 配置**:
```bash
python -m sglang.launch_server \
    --model-path Qwen/Qwen3-32B \
    --port 30000 \
    --dtype bfloat16 \
    --chunked-prefill-size 4096 \
    --max-prefill-tokens 16384 \
    --mem-fraction-static 0.85 \
    --max-running-requests 128
```

**H20 配置**:
```bash
python -m sglang.launch_server \
    --model-path Qwen/Qwen3-32B \
    --port 30010 \
    --dtype bfloat16 \
    --chunked-prefill-size 2048 \
    --max-prefill-tokens 8192 \
    --mem-fraction-static 0.88 \
    --max-running-requests 192
```

### 4.4 Chunked Prefill 的优势

| 优势 | 说明 | 效果 |
|------|------|------|
| **降低 TTFT 尖峰** | 长 prompt 不再阻塞所有 Decode 请求 | TTFT P99 降低 40-60% |
| **减少显存峰值** | 分块处理,无需一次性存储全部 KV Cache | 显存峰值降低 30% |
| **提升 GPU 利用率** | Prefill 和 Decode 交错执行,避免资源闲置 | 吞吐量提升 15-25% |
| **避免 OOM** | 超长 prompt (如 50K tokens) 也能处理 | 支持更长上下文 |

### 4.5 调优建议

**Chunk Size 选择**:

| 场景 | 推荐 Chunk Size | 理由 |
|------|----------------|------|
| 显存充裕 (剩余 > 20GB) | 8192 | 减少调度开销 |
| 显存紧张 (剩余 10-20GB) | 4096 | 平衡性能和显存 |
| 显存紧张 (剩余 < 10GB) | 2048 | 优先避免 OOM |
| 混合负载 (长短请求并存) | 4096 | 通用最优值 |

## 五、关键技术:Continuous Batching

### 5.1 什么是 Continuous Batching?

**传统 Batching 的问题**:
- **Static Batching**: 等待 batch 满后再执行,短请求被长请求阻塞
- **Head-of-Line Blocking**: 队头请求完成时,才能添加新请求

**Continuous Batching 解决方案**:
- 请求**动态进入和退出 batch**,无需等待整个 batch 完成
- 已完成的请求立即被新请求替换,保持 GPU 高利用率
- 消除队头阻塞,降低延迟

### 5.2 工作机制

```
传统 Static Batching:
Batch 1: [Req1(长), Req2(短), Req3(短), Req4(短)]
  ↓
  等待所有请求完成...
  ↓ (Req2/3/4 早已完成,但必须等待 Req1)
  全部完成,释放 batch
  ↓
Batch 2: [Req5, Req6, Req7, Req8]
  ↓
  大量时间浪费在等待上!

Continuous Batching:
Batch: [Req1(长), Req2(短), Req3, Req4]
  ↓
  Req2 完成 → 立即移除,添加 Req5
  ↓
Batch: [Req1(长), Req5, Req3, Req4]
  ↓
  Req3 完成 → 立即移除,添加 Req6
  ↓
Batch: [Req1(长), Req5, Req6, Req4]
  ↓
  GPU 始终满载,无等待时间!
```

### 5.3 SGLang 配置

**SGLang 默认使用 Continuous Batching**,无需额外配置!

**关键参数**:

| 参数 | 说明 | 默认值 | H800 推荐 | H20 推荐 |
|------|------|--------|----------|---------|
| `--max-running-requests` | 最大并发活跃请求数 | 自动计算 | **128** | **192** |
| `--schedule-policy` | 调度策略 | `fcfs` | **`lpm`** | **`fcfs`** |
| `--schedule-conservativeness` | 调度保守程度 | 1.0 | **0.6** | **0.8** |

**调度策略选择**:

| 策略 | 全称 | 工作原理 | 适用场景 |
|------|------|----------|---------|
| `fcfs` | First Come First Served | 按到达顺序调度 | 简单负载,无缓存复用 |
| `lpm` | Longest Prefix Match | 优先调度与 KV Cache 匹配度最高的请求 | 多轮对话,有共享前缀 |
| `lof` | Longest Output First | 优先处理最大生成长度的请求 | 批量推理,追求吞吐 |

### 5.4 单卡利用率优化

**H800 优化配置**:
```bash

    --model-path Qwen/Qwen3-32B \
    --port 30000 \
    --dtype bfloat16 \

    --max-running-requests 128 \
```

**参数解释**:

- `--max-running-requests 128`: 最多 128 个并发请求
- `--schedule-conservativeness 0.6`: 适度激进调度 (显存充裕)

**H20 优化配置**:
```bash
python -m sglang.launch_server \
    --model-path Qwen/Qwen3-32B \
    --port 30010 \
    --dtype bfloat16 \
    --mem-fraction-static 0.88 \
    --max-running-requests 192 \
```

**参数解释**:
- `--mem-fraction-static 0.88`: KV Cache 池占 88% 显存 (约 84GB)
- `--max-running-requests 192`: 最多 192 个并发请求 (显存更大)
- `--schedule-conservativeness 0.8`: 适度保守 (算力弱,避免 OOM)
- `--schedule-policy fcfs`: 简单调度 (算力弱,减少 overhead)

### 5.5 Overlapped Scheduler (零开销调度)

**原理**: GPU 执行当前 batch 时,CPU 异步准备下一批

```
时间线:
┌──────────────────────────────────────────────┐
│ GPU: 执行 Batch 1 (100ms)                     │
└──────────────────────────────────────────────┘
         ↓ (同时)
┌──────────────────────────────────────────────┐
│ CPU: 准备 Batch 2                             │
│   - RadixAttention 前缀匹配                    │
│   - 内存分配                                  │
│   - 语法约束准备                               │
└──────────────────────────────────────────────┘
         ↓
┌──────────────────────────────────────────────┐
│ GPU: 立即执行 Batch 2 (无等待!)                │
└──────────────────────────────────────────────┘
```

**效果**: 吞吐量提升 **10-20%**

**启用方式**: SGLang **默认启用**,无需额外参数!

## 六、策略选型分析

### 6.1 候选策略对比

在 PD 不分离架构下,我们有 **2 个独立 Worker** (H800 和 H20),可以做的路由策略:

| 策略 | 适配度 | 理由 | 评分 |
|------|-------|------|------|
| **RequestClassification** | ⭐⭐⭐⭐⭐ | 根据请求特征智能路由,完美适配异构 | **9.5/10** |
| **PerformanceAware** | ⭐⭐⭐⭐⭐ | 基于实际性能动态选择,长期最优 | **9/10** |
| **RequestSizeBucket** | ⭐⭐⭐⭐ | 只看输入长度,简单有效 | **7.5/10** |
| **PowerOfTwo + Load** | ⭐⭐⭐ | 只考虑负载,忽略硬件差异 | **6/10** |
| **RoundRobin** | ⭐⭐ | 完全忽略硬件差异 | **4/10** |
| **CacheAware** | ⭐⭐⭐ | 有缓存复用时有用 | **6.5/10** |

### 6.2 🏆 推荐方案:RequestClassification (请求分类)

**理由**:

1. **异构硬件差异明显**: H800 算力 3 倍强,H20 显存/带宽优势明显
2. **请求特征可预估**: input_length 和 max_tokens 可从请求中获取
3. **路由逻辑清晰**: 计算密集型 → H800,内存密集型 → H20
4. **实测验证**: 之前测试成功率 100%,分类准确

**分类逻辑**:

| 请求类型 | 输入长度 | 输出长度 | 典型场景 | 路由目标 | 理由 |
|---------|---------|---------|---------|---------|------|
| **计算密集型** | < 100 tokens | > 100 tokens | 创意写作、代码生成、翻译 | **H800** | 需要高算力快速生成 |
| **内存密集型** | > 500 tokens | < 500 tokens | 文档摘要、问答、分类 | **H20** | 需要大显存存储长上下文 |
| **均衡型** | 100-500 tokens | 100-500 tokens | 对话、中等文本生成 | **负载较低者** | 两者都能胜任 |

**配置示例**:

```json
{
  "policy": {
    "type": "request_classification",
    "short_input_threshold": 100,
    "medium_input_threshold": 500,
    "small_output_threshold": 100,
    "medium_output_threshold": 500,
    "auto_assign_workers": true
  }
}
```

**Gateway 启动命令**:

```bash
cd /path/to/sgl-model-gateway

./target/debug/sgl-model-gateway \
    --worker http://127.0.0.1:30000 \
    --worker http://127.0.0.1:30010 \
    --host 0.0.0.0 \
    --port 3000 \
    --policy request_classification \
    --short-input-threshold 100 \
    --medium-input-threshold 500 \
    --small-output-threshold 100 \
    --medium-output-threshold 500 \
    --auto-assign-workers true \
    --log-level info
```

### 6.3 备选方案:PerformanceAware (性能感知)

**适用条件**: 
- 运行一段时间后有足够性能数据
- 希望自适应硬件性能波动
- 不想手工调优分类阈值

**工作原理**:
- 收集每个 Worker 的 TTFT、TPOT、吞吐量指标
- 归一化后计算加权分数
- 选择分数最高的 Worker

**配置示例**:

```json
{
  "policy": {
    "type": "performance_aware",
    "weight_ttft": 0.3,
    "weight_tpot": 0.4,
    "weight_throughput": 0.3,
    "score_refresh_interval_secs": 60,
    "consider_load": true
  }
}
```

**优缺点对比**:

| 维度 | RequestClassification | PerformanceAware |
|------|----------------------|------------------|
| **冷启动** | ✅ 立即可用 | ⚠️ 需要积累数据 |
| **可解释性** | ✅ 规则清晰 | ❌ 黑盒评分 |
| **自适应** | ❌ 需手工调优 | ✅ 自动适应 |
| **准确度** | ✅ 高 (基于理论) | ✅ 高 (基于实测) |
| **复杂度** | 低 | 中 |
| **推荐场景** | 初期部署 | 长期运行优化 |

**建议**: **初期用 RequestClassification,稳定后切换到 PerformanceAware**。

### 6.4 备选方案:RequestSizeBucket (请求长度分桶)

**适用条件**: 
- 无法获取 max_tokens (输出长度未知)
- 只关心输入长度
- 需要简单方案

**分类逻辑**:

```
输入 < 100 tokens    → H800 (短请求,算力浪费但延迟低)
输入 100-500 tokens  → 负载较低者
输入 > 500 tokens    → H20 (长请求,大显存优势)
```

**配置示例**:

```json
{
  "policy": {
    "type": "request_size_bucket",
    "short_threshold": 100,
    "medium_threshold": 500,
    "track_load_per_bucket": true
  }
}
```

**为什么不推荐**: 只看输入长度,忽略输出长度,路由决策不够精准。

### 6.5 策略组合建议

**推荐组合**: **RequestClassification + SessionAffinity**

```
场景: 多轮对话应用

第一轮请求:
  → RequestClassification 路由到 H800

后续请求 (同一会话):
  → SessionAffinity 保持路由到 H800
  → 利用 KV Cache,避免上下文迁移
```

**配置示例**:

```json
{
  "policy": {
    "type": "session_affinity",
    "timeout_minutes": 30,
    "fallback_policy": {
      "type": "request_classification",
      "short_input_threshold": 100,
      "medium_input_threshold": 500,
      "small_output_threshold": 100,
      "medium_output_threshold": 500,
      "auto_assign_workers": true
    }
  }
}
```

### 7.2 H800 启动脚本

```bash
#!/bin/bash

MODEL_PATH="/path/to/Qwen/Qwen3-32B"
PORT=30000

echo "Starting H800 server on port $PORT..."

    --model-path "$MODEL_PATH" \
```

## 八、性能优化指南

### 8.1 关键参数调优路径

#### Chunked Prefill Size

```
调优步骤:
1. 监控显存使用: nvidia-smi -l 1
2. 如果显存使用率 < 70%: 增大 chunked-prefill-size (减少调度开销)
3. 如果显存使用率 > 90%: 减小 chunked-prefill-size (避免 OOM)
4. 观察 TTFT P99: 如果降低,说明 chunk size 合适
```

**推荐值**:

| GPU | 显存 | 初始值 | 调优方向 |
|-----|------|-------|---------|
| H800 | 80GB | 4096 | 显存充裕 → 8192 |
| H20 | 96GB | 2048 | 显存充裕 → 4096 |

#### Max Running Requests

```
调优步骤:
1. 压测,逐步增加并发数 (32 → 64 → 128 → 192 → 256)
2. 观察 P99 延迟:
   - 如果延迟稳定: 继续增加
   - 如果延迟飙升: 回退到上一个值
3. 找到"延迟可接受"的最大并发数
```

**推荐值**:

| GPU | 初始值 | 延迟敏感场景 | 吞吐敏感场景 |
|-----|-------|------------|------------|
| H800 | 128 | 64-96 | 128-192 |
| H20 | 192 | 96-128 | 192-256 |

#### Schedule Conservativeness

```
调优步骤:
1. 默认 1.0 (保守)
2. 如果频繁 OOM: 增加到 1.3
3. 如果显存充裕但吞吐低: 降低到 0.6-0.8 (更激进)
4. 观察 pending queue 深度:
   - 持续积压 → 降低 conservativeness
   - 频繁 OOM → 提高 conservativeness
```

### 8.2 量化策略 (可选)

**场景**: 如果 H20 显存不足以运行 BF16 全精度,或对延迟要求不高

| GPU | 精度 | 显存占用 | 性能影响 | 适用场景 |
|-----|------|---------|---------|---------|
| H800 | BF16 | ~78GB | 基线 | 计算密集型请求 |
| H20 | BF16 | ~78GB | 基线 | 内存密集型请求 |
| H20 | INT8 | ~42GB | 延迟 +10-20% | 显存紧张时 |
| H20 | FP8 | ~42GB | 延迟 +5-15% | H20 推荐量化 |

**H20 INT8 量化配置**:

```bash
python -m sglang.launch_server \
    --model-path Qwen/Qwen3-32B \
    --port 30010 \
    --quantization fp8 \
    --mem-fraction-static 0.88 \
    --max-running-requests 256 \
    --chunked-prefill-size 2048
```

**效果**:
- 显存占用从 78GB → 42GB (节省 46%)
- 可支持更多并发 (256 vs 192)
- 延迟增加 5-15% (可接受)

### 8.3 监控指标

**关键指标**:

| 指标 | 说明 | 告警阈值 | 优化策略 |
|------|------|---------|---------|
| **GPU 显存使用率** | `nvidia-smi` 查看 | > 95% | 降低 max-running-requests 或 chunk size |
| **GPU 利用率** | `nvidia-smi` 查看 | < 50% | 提高 max-running-requests |
| **TTFT P99** | 首 Token 延迟 P99 | > 2s | 减小 chunk size,增加 H800 并发 |

| **Pending Queue** | 等待队列深度 | > 1000 | 增加并发数或优化路由 |
| **OOM 次数** | 显存溢出次数 | > 0 | 降低 mem-fraction 或 conservativeness |

## 十、风险与应对

### 10.1 潜在风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| **分类阈值不当** | 路由不均,某 GPU 过载 | 中 | 压测调优,监控告警 |
| **输出长度预估不准** | RequestClassification 分类错误 | 高 | 使用默认值,或训练预估模型 |
| **H20 算力瓶颈** | 计算密集型请求延迟高 | 中 | 严格路由,避免 H20 处理计算密集型 |
| **显存溢出 (OOM)** | 请求失败,服务中断 | 低 | 降低 mem-fraction 或 max-running-requests |
| **Chunk size 不当** | TTFT 高或显存浪费 | 中 | 根据显存使用率动态调整 |
| **单点故障** | H800 或 H20 故障,损失 50% 容量 | 低 | 监控告警,快速重启 |

|------|------|
| **架构模式** | PD 不分离 (2 独立 Worker) |
| **路由策略** | RequestClassification (初期) → PerformanceAware (长期) |
| **关键技术** | Chunked Prefill + Continuous Batching |
| **H800 角色** | 计算密集型请求 (短输入 + 长输出) |
| **H20 角色** | 内存密集型请求 (长输入 + 短输出) |
| **量化策略** | H800: BF16, H20: BF16 (或 INT8 显存紧张时) |

### 11.2 核心优势

1. **有调度空间**: 2 个独立 Worker,可以智能路由
2. **部署简单**: 无需 PD 分离的 Bootstrap 传输
3. **故障隔离**: 单卡故障不影响另一卡
4. **性能优化**: Chunked Prefill + Batching 提升单卡利用率
5. **易于扩展**: 未来增加 GPU 时,直接加入路由池
