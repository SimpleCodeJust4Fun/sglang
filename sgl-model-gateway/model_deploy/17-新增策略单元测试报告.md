## 3. 测试详细结果

### 3.1 RequestSizeBucket 策略测试

**测试文件**: `src/policies/request_size_bucket.rs`
**策略说明**: 根据请求输入长度 (字符数) 将请求分类到不同桶 (Short/Medium/Long),
并路由到对应的优化 Worker。

#### 测试结果

| 测试用例 | 状态 | 耗时 | 说明 |
|----------|------|------|------|
| `test_request_classification` | ✅ PASS | < 0.03s | 验证请求按长度正确分类到 Short/Medium/Long 桶 |
| `test_worker_assignment` | ✅ PASS | < 0.03s | 验证 Worker 根据 priority/cost 正确分配到不同桶 |
| `test_routing_by_request_size` | ✅ PASS | < 0.03s | 验证路由逻辑能正确选择低负载的合适 Worker |

#### 测试覆盖场景

1. **分类逻辑**:
   - Short: char_count < short_threshold (默认 100)
   - Medium: short_threshold <= char_count < medium_threshold (默认 500)
   - Long: char_count >= medium_threshold

2. **Worker 分配**:
   - 高 priority + 低 cost → Short 桶 (低延迟优化)
   - 中等 priority/cost → Medium 桶 (均衡型)
   - 低 priority + 高 cost → Long 桶 (大内存型)

3. **路由选择**:
   - 在同一桶内选择负载最低的 Worker
   - 使用随机选择作为负载均衡的 fallback

### 3.2 PerformanceAware 策略测试

**测试文件**: `src/policies/performance_aware.rs`
**策略说明**: 基于 Worker 性能指标 (TTFT/TPOT/Tokens-per-sec) 进行智能路由,
适用于异构 GPU 环境。

#### 测试结果

| 测试用例 | 状态 | 耗时 | 说明 |
|----------|------|------|------|
| `test_metrics_update` | ✅ PASS | < 0.03s | 验证性能指标更新逻辑正确 |
| `test_performance_aware_no_metrics_fallback` | ✅ PASS | < 0.03s | 验证无指标时 fallback 到随机选择 |
| `test_performance_aware_scoring` | ✅ PASS | < 0.03s | 验证性能评分计算逻辑正确 |

#### 测试覆盖场景

1. **指标收集**:
   - TTFT (Time To First Token): 首 token 延迟
   - TPOT (Time Per Output Token): 每 token 生成时间
   - Tokens/sec: 吞吐量指标

2. **评分计算**:
   - 加权评分公式: `score = w_ttft * norm_ttft + w_tpot * norm_tpot + w_throughput * norm_throughput`
   - 默认权重: TTFT=0.3, TPOT=0.3, Throughput=0.4
   - 指标归一化到 0-1 范围 (TTFT/TPOT 取反,越低越好)

3. **Fallback 机制**:
   - 无性能指标时随机选择健康 Worker
   - 保证系统在任何状态下都能正常工作

### 3.3 RequestClassification 策略测试

**测试文件**: `src/policies/request_classification.rs`
**策略说明**: 多维度请求分类策略,综合考虑输入长度、输出长度和计算模式,
将请求路由到最适合的 Worker 类型。

#### 测试结果

| 测试用例 | 状态 | 耗时 | 说明 |
|----------|------|------|------|
| `test_request_classification_compute_intensive` | ✅ PASS | < 0.02s | 验证计算密集型请求正确分类 |
| `test_request_classification_memory_intensive` | ✅ PASS | < 0.02s | 验证内存密集型请求正确分类 |
| `test_routing_with_classification` | ✅ PASS | < 0.02s | 验证分类后路由选择逻辑 |
| `test_worker_assignment` | ✅ PASS | < 0.02s | 验证 Worker 按类型正确分配 |

#### 测试覆盖场景

1. **多维度分类**:
   - 输入长度: Short (<100), Medium (100-500), Long (>500 chars)
   - 输出长度: Small (<100), Medium (100-500), Large (>500 tokens)
   - 计算模式:
     - Compute-intensive: 短输入 + 长输出 (如创意写作)
     - Memory-intensive: 长输入 + 短输出 (如文本摘要)
     - Balanced: 中等输入 + 中等输出

2. **Worker 类型分配**:
   - 高端 GPU (快速计算) → Compute-intensive 请求
   - 大内存 GPU → Memory-intensive 请求
   - 均衡型 GPU → Balanced 请求

3. **路由逻辑**:
   - 根据请求分类选择对应类型的 Worker
   - 在同类型 Worker 中选择负载最低的

## 6. 已知问题

### 6.1 Rust 编译器 ICE (Internal Compiler Error)

**问题描述**: Rust 1.94.1 编译器在处理 `policies::request_classification` 模块的
未使用变量检查时会发生 panic。

**错误信息**:
```
thread 'rustc' panicked at ... slice index starts at 14 but ends at 12
query stack during panic:
#0 [check_mod_deathness] checking deathness of variables in module `policies::request_classification`
#1 [analysis] running analysis passes on crate `smg`
```

**临时解决方案**: 使用 `RUSTFLAGS='-A dead_code -A unused'` 禁用相关 lint 检查。

**建议**: 考虑升级 Rust 版本或向 Rust 项目提交 bug 报告。
