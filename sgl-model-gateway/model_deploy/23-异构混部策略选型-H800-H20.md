# 异构混部策略选型报告 - H800+H20 部署 Qwen3 30B

**文档版本**: 1.0  
**创建日期**: 2026-04-20  
**适用场景**: H800 (80GB) + H20 (96GB) 异构 GPU 混部  
**模型**: Qwen3 30B (约 60GB FP16 / 30GB INT8)

---

## 一、项目目标与场景分析

### 1.1 硬件环境

| GPU 型号 | 显存 | 算力 (FP16) | 带宽 | 数量 | 特点 |
|---------|------|------------|------|------|------|
| **H800** | 80GB HBM2e | 31 TFLOPS | 2.0 TB/s | N 卡 | 高算力,适合 Prefill |
| **H20** | 96GB HBM3 | 10 TFLOPS | 4.0 TB/s | M 卡 | 大显存,高带宽,适合 Decode |

**关键差异**:
- **H800**: 算力是 H20 的 **3 倍**,但显存少 20%
- **H20**: 显存大 20%,带宽是 H800 的 **2 倍**,但算力只有 1/3

### 1.2 工作负载特征

Qwen3 30B 模型的典型请求模式:

| 请求类型 | 输入 Tokens | 输出 Tokens | 计算特点 | 占比 |
|---------|-------------|-------------|---------|------|
| **短对话** | 10-100 | 10-100 | Prefill 短,Decode 短 | 30% |
| **问答** | 100-1000 | 100-500 | Prefill 中等,Decode 中等 | 40% |
| **长文本生成** | 100-500 | 1000-4000 | Prefill 短,Decode 长 | 20% |
| **文档分析** | 4000-32000 | 500-2000 | Prefill 长,Decode 中等 | 10% |

### 1.3 性能瓶颈分析

**Prefill 阶段** (输入处理):
- **瓶颈**: 算力 (TFLOPS)
- **内存需求**: 输入 KV Cache ~ O(input_tokens × hidden_size)
- **H800 优势**: 3 倍算力,Prefill 速度快 2-3 倍

**Decode 阶段** (输出生成):
- **瓶颈**: 内存带宽 (逐 token 生成)
- **内存需求**: 全量 KV Cache ~ O(output_tokens × hidden_size)
- **H20 优势**: 2 倍带宽,Decode 吞吐量高 1.5-2 倍

**结论**: 这是典型的 **异构计算场景**,需要 PD 分离 + 智能调度!

## 二、架构设计建议

### 2.1 推荐架构:PD 分离 + 异构调度

```
                    ┌─────────────────────────────┐
                    │   Gateway (策略路由)         │
                    │   Host: 127.0.0.1:3000      │
                    └──────────┬──────────────────┘
                               │
                    ┌──────────┴──────────┐
                    │   Prefill 层        │
                    │   Policy: ?         │
                    │   GPU: H800 × 2    │
                    ├─────────────────────┤
                    │ P1: H800 :30000    │
                    │ P2: H800 :30001    │
                    └──────────┬──────────┘
                               │ KV Cache Bootstrap
                    ┌──────────┴──────────┐
                    │   Decode 层         │
                    │   Policy: ?         │
                    │   GPU: H20 × 2     │
                    ├─────────────────────┤
                    │ D1: H20  :30010    │
                    │ D2: H20  :30011    │
                    └─────────────────────┘
```

### 2.2 GPU 分配策略

| 层 | GPU 型号 | 数量 | 单卡显存分配 | 总显存 | 用途 |
|----|---------|------|------------|--------|------|
| **Prefill** | H800 | 2 | 70GB (87%) | 140GB | 处理输入,Prefill 计算 |
| **Decode** | H20 | 2 | 85GB (88%) | 170GB | 生成输出,维护 KV Cache |

**为什么这样分配?**
- **Prefill 用 H800**: 算力密集,需要快速处理输入
- **Decode 用 H20**: 内存带宽密集,大显存可容纳更多并发请求的 KV Cache

## 三、策略选型分析

### 3.1 Prefill 层策略选型

#### 候选策略对比

| 策略 | 适配度 | 理由 | 评分 |
|------|-------|------|------|
| **Cache Aware** | ⭐⭐⭐ | Prefill 阶段短,缓存收益低 | 6/10 |
| **Power of Two** | ⭐⭐⭐⭐ | 简单有效,考虑负载 | 7/10 |
| **Request Size Bucket** | ⭐⭐⭐⭐⭐ | 根据输入长度路由,完美适配 | 9/10 |
| **Performance Aware** | ⭐⭐⭐⭐ | 自适应 H800 性能差异 | 8/10 |
| **Round Robin** | ⭐⭐⭐ | 简单均匀,不考虑特征 | 6/10 |

#### 🏆 推荐方案:Request Size Bucket

**理由**:
1. **Prefill 性能与输入长度强相关**: 输入越长,计算量越大
2. **H800 算力充足但显存有限**: 需要根据输入长度分流,避免单卡 KV Cache 溢出
3. **实现简单,可预测**: 阈值调优后效果稳定

**配置建议**:
```bash
--prefill-policy request_size_bucket \
  --short-threshold 500 \        # <500 tokens 为短请求
  --medium-threshold 4000 \      # 500-4000 tokens 为中等
  --track-load-per-bucket        # 分别跟踪每个桶的负载
```

**路由逻辑**:
```
输入 < 500 tokens    → 随机分配到 P1/P2 (都轻松处理)
输入 500-4000 tokens → 选择负载较低的 Prefill 节点
输入 > 4000 tokens   → 选择 KV Cache 空间更充足的节点
```

#### 备选方案:Performance Aware

**适用条件**: 如果两张 H800 实际性能有差异 (如散热降频、共享总线带宽竞争)

**配置建议**:
```bash
--prefill-policy performance_aware \
  --weight-ttft 0.5 \              # Prefill 关注 TTFT
  --weight-tpot 0.3 \
  --weight-throughput 0.2 \
  --score-refresh-interval-secs 60
```

### 3.2 Decode 层策略选型

#### 候选策略对比

| 策略 | 适配度 | 理由 | 评分 |
|------|-------|------|------|
| **Cache Aware** | ⭐⭐⭐⭐⭐ | 最大化 KV Cache 命中,Decode 收益高 | 9/10 |
| **Power of Two** | ⭐⭐⭐⭐ | 考虑负载,简单有效 | 8/10 |
| **Request Classification** | ⭐⭐⭐⭐ | 多维度分类,适配不同输出模式 | 8/10 |
| **Performance Aware** | ⭐⭐⭐⭐⭐ | 自适应 H20 性能,关注吞吐量 | 9/10 |
| **Round Robin** | ⭐⭐⭐ | 均匀但不智能 | 6/10 |

#### 🏆 推荐方案:Performance Aware

**理由**:
1. **Decode 是持续性能敏感阶段**: 每个 token 生成速度直接影响用户体验
2. **H20 大显存优势**: 可容纳更多并发请求,需要根据实际吞吐量调度
3. **自适应长期运行**: 性能数据采样后自动调整,无需手工调优

**配置建议**:
```bash
--decode-policy performance_aware \
  --weight-ttft 0.2 \              # Decode 阶段 TTFT 权重低
  --weight-tpot 0.5 \              # TPOT 最重要 (逐 token 生成)
  --weight-throughput 0.3 \        # 整体吞吐量
  --score-refresh-interval-secs 60 \
  --consider-load true             # 考虑当前负载
```

**评分公式针对 Decode 优化**:
```
decode_score = (0.2 × norm_TTFT + 0.5 × norm_TPOT + 0.3 × norm_TokensPerSec) × load_factor
```

#### 备选方案 1:Cache Aware (会话重复率高时)

**适用条件**: 如果业务场景有大量相似请求 (如客服、FAQ)

**配置建议**:
```bash
--decode-policy cache_aware \
  --cache-threshold 0.3 \          # 30% 前缀匹配即路由到该节点
  --balance-abs-threshold 64 \     # 绝对负载差异阈值
  --balance-rel-threshold 1.5      # 相对负载比例阈值
```

**优势**: 如果多个用户问相似问题,KV Cache 命中可提速 2-3 倍

#### 备选方案 2:Request Classification (混合负载场景)

**适用条件**: 如果请求类型多样 (短对话 + 长文生成混合)

**配置建议**:
```bash
--decode-policy request_classification \
  --short-input-threshold 500 \
  --medium-input-threshold 4000 \
  --small-output-threshold 500 \
  --medium-output-threshold 2000 \
  --auto-assign-workers true
```

**路由逻辑**:
```
ComputeIntensive (短输入+长输出)  → 算力更强的 Decode 节点
MemoryIntensive (长输入+短输出)  → 显存更大的 Decode 节点
Balanced (中等输入+中等输出)     → 负载均衡分配
```

### 3.3 非 PD 分离模式策略选型

如果不使用 PD 分离,所有 GPU 混合服务:

#### 🏆 推荐方案:Request Classification

**配置建议**:
```bash
--policy request_classification \
  --short-input-threshold 500 \
  --medium-input-threshold 4000 \
  --small-output-threshold 500 \
  --medium-output-threshold 2000 \
  --auto-assign-workers true
```

**路由逻辑**:
```
ComputeIntensive 请求 (短输入+长输出)
  → 优先路由到 H800 (算力强,生成快)

MemoryIntensive 请求 (长输入+短输出)
  → 优先路由到 H20 (显存大,带宽高)

Balanced 请求
  → 根据负载均匀分配
```

**Worker 标签设置**:
```bash
# H800 节点启动
python3 -m sglang.launch_server \
  --model-path Qwen3-30B \
  --port 30000 \
  --mem-fraction-static 0.87 \
  --host 127.0.0.1
  # Gateway 通过性能自动识别为"高算力"节点

# H20 节点启动
python3 -m sglang.launch_server \
  --model-path Qwen3-30B \
  --port 30010 \
  --mem-fraction-static 0.88 \
  --host 127.0.0.1
  # Gateway 通过性能自动识别为"大显存"节点
```

## 四、完整启动配置示例

### 4.1 PD 分离模式 (推荐)

#### 启动 Prefill 节点 (H800)

```bash
#!/bin/bash
# start-prefill-h800.sh

MODEL_PATH="/path/to/Qwen3-30B"
MEM_FRACTION=0.87  # 70GB / 80GB
CONTEXT_LENGTH=32768

# Prefill-1 on H800-1
setsid python3 -m sglang.launch_server \
    --model-path "$MODEL_PATH" \
    --port 30000 \
    --mem-fraction-static $MEM_FRACTION \
    --tp 1 \
    --pd prefill \
    --disaggregation-bootstrap-port 9000 \
    --host 127.0.0.1 \
    --context-length $CONTEXT_LENGTH \
    --log-level info \
    > /tmp/sglang-prefill-1.log 2>&1 &

# Prefill-2 on H800-2
setsid python3 -m sglang.launch_server \
    --model-path "$MODEL_PATH" \
    --port 30001 \
    --mem-fraction-static $MEM_FRACTION \
    --tp 1 \
    --pd prefill \
    --disaggregation-bootstrap-port 9001 \
    --host 127.0.0.1 \
    --context-length $CONTEXT_LENGTH \
    --log-level info \
    > /tmp/sglang-prefill-2.log 2>&1 &

echo "Prefill nodes starting..."
```

#### 启动 Decode 节点 (H20)

```bash
#!/bin/bash
# start-decode-h20.sh

MODEL_PATH="/path/to/Qwen3-30B"

    --pd decode \
    --host 127.0.0.1 \
    --context-length $CONTEXT_LENGTH \
    --log-level info \
    > /tmp/sglang-decode-1.log 2>&1 &

# Decode-2 on H20-2
setsid python3 -m sglang.launch_server \
    --model-path "$MODEL_PATH" \
    --port 30011 \
    --mem-fraction-static $MEM_FRACTION \
    --tp 1 \
    --pd decode \
    --host 127.0.0.1 \
    --context-length $CONTEXT_LENGTH \
    --log-level info \
    > /tmp/sglang-decode-2.log 2>&1 &

```

#### 启动 Gateway

```bash
#!/bin/bash
# start-gateway.sh

cd /path/to/sgl-model-gateway

./target/debug/sgl-model-gateway \
    --pd-disaggregation \
    --prefill http://127.0.0.1:30000 9000 \
    --prefill http://127.0.0.1:30001 9001 \
    --decode http://127.0.0.1:30010 \
    --decode http://127.0.0.1:30011 \
    --host 0.0.0.0 \
    --port 3000 \
    --prefill-policy request_size_bucket \
    --short-threshold 500 \
    --medium-threshold 4000 \
    --track-load-per-bucket \
    --decode-policy performance_aware \
    --weight-ttft 0.2 \
    --weight-tpot 0.5 \
    --weight-throughput 0.3 \
    --score-refresh-interval-secs 60 \
    --consider-load true \
    --log-level info

echo "Gateway started on port 3000"
```

## 五、性能优化建议

### 5.1 阈值调优

**初始值** (需根据实际压测调整):

| 参数 | 初始值 | 调优方向 | 观察指标 |
|------|-------|---------|---------|
| `short_threshold` | 500 tokens | 根据 P50 输入长度调整 | 短请求占比 |
| `medium_threshold` | 4000 tokens | 根据 P90 输入长度调整 | 长请求溢出率 |
| `weight_tpot` | 0.5 | 如果延迟敏感,提高到 0.6-0.7 | P99 延迟 |
| `weight_throughput` | 0.3 | 如果吞吐敏感,提高到 0.4-0.5 | tokens/sec |

## 六、策略迁移路径

### 6.1 分阶段部署

| 阶段 | Prefill 策略 | Decode 策略 | 目标 |
|------|-------------|------------|------|
| **Phase 1** (第 1 周) | Round Robin | Round Robin | 基线性能 |
| **Phase 2** (第 2 周) | Request Size Bucket | Round Robin | 优化 Prefill |
| **Phase 3** (第 3 周) | Request Size Bucket | Performance Aware | 全链路优化 |
| **Phase 4** (第 4 周+) | Performance Aware | Performance Aware | 自适应调优 |

### 6.2 A/B 测试

**对比实验**:
```bash
# 实验 A: 当前推荐方案
--prefill-policy request_size_bucket
--decode-policy performance_aware

# 实验 B: 全缓存方案
--prefill-policy cache_aware
--decode-policy cache_aware

# 实验 C: 全性能方案
--prefill-policy performance_aware
--decode-policy performance_aware
```

**观察周期**: 每个方案运行 24-48 小时,对比:
- P50/P95/P99 延迟
- 吞吐量 (tokens/sec)
- GPU 利用率
- KV Cache 命中率

## 七、风险与应对

### 7.1 潜在风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| **阈值配置不当** | 负载不均 | 中 | 压测调优,监控告警 |
| **Performance Aware 冷启动** | 初期性能数据不足 | 高 | Phase 1 用 Round Robin 预热 |
| **H800 显存溢出** | 长请求 OOM | 低 | Request Size Bucket 分流 |
| **H20 算力瓶颈** | Decode 延迟高 | 中 | 增加 H20 节点数量 |
| **网络带宽瓶颈** | KV Cache 传输慢 | 低 | 使用 NVLink/InfiniBand |

### 7.2 降级方案

**如果策略效果不佳,可快速降级**:
```bash
# 降级到最简单的负载均衡
--prefill-policy round_robin
--decode-policy round_robin

# 或使用 Power of Two (简单有效)
--prefill-policy power_of_two
--decode-policy power_of_two
```

## 八、总结与建议

### 8.1 最终推荐

| 场景 | Prefill 策略 | Decode 策略 | 理由 |
|------|-------------|------------|------|
| **生产环境 (推荐)** | Request Size Bucket | Performance Aware | 简单+智能,最佳平衡 |
| **会话重复率高** | Cache Aware | Cache Aware | 最大化缓存命中 |
| **快速上线** | Round Robin | Round Robin | 零配置,均匀分布 |
| **极致性能** | Performance Aware | Performance Aware | 全自适应,长期最优 |

### 8.2 核心原则

1. **Prefill 重算力,Decode 重带宽** → PD 分离是异构混部的关键
2. **根据输入特征路由** → Request Size Bucket / Classification 比纯负载感知更有效
3. **长期运行用 Performance Aware** → 自适应硬件差异,无需手工调优
4. **缓存对 Decode 更重要** → 如果业务有重复模式,优先 Cache Aware
