## 1. 实验目标

验证三种新增的异构 GPU 调度策略在真实 PD 分离架构中的功能:

1. **RequestSizeBucket**: 根据请求输入长度分类路由
2. **PerformanceAware**: 基于 Worker 性能评分路由
3. **RequestClassification**: 多维度请求特征分类路由

## 4. 实验结果

### 4.1 RequestSizeBucket 策略

**测试结果**: ✅ 全部成功

| 请求类型 | HTTP 状态 | 延迟 (ms) | Prompt Tokens | Completion Tokens | Total Tokens |
|----------|-----------|-----------|---------------|-------------------|--------------|
| **Short** | 200 | 362 | 31 | 10 | 41 |
| **Medium** | 200 | 1372 | 67 | 200 | 267 |
| **Long** | 200 | 3019 | 147 | 500 | 647 |

**分析**:
- Short 请求 (< 100 chars) 被正确分类到 Short bucket
- Medium 请求 (100-500 chars) 被正确分类到 Medium bucket
- Long 请求 (> 500 chars) 被正确分类到 Long bucket
- 延迟随请求长度增加而增加,符合预期
- Gateway 启动和 Worker 发现正常

**路由决策**:
- 策略根据 Worker 的 priority/cost 元数据自动分配到不同 bucket
- 由于是单 GPU 模拟,所有 Worker 性能相似,路由决策主要基于负载

### 4.2 PerformanceAware 策略

**测试结果**: ✅ 全部成功

| 请求类型 | HTTP 状态 | 延迟 (ms) | Prompt Tokens | Completion Tokens | Total Tokens |

| **Short** | 200 | 395 | 31 | 10 | 41 |
| **Medium** | 200 | 1340 | 67 | 200 | 267 |
| **Long** | 200 | 2994 | 147 | 500 | 647 |

**分析**:
- 性能评分机制正常工作

- 随着请求增多,性能评分会逐渐准确,路由决策会更优化

**评分机制**:
- 默认权重: TTFT=0.3, TPOT=0.3, Throughput=0.4
- 评分刷新间隔: 60 秒

## 5. 性能分析

### 5.1 延迟对比

| 策略 | Short (ms) | Medium (ms) | Long (ms) | 平均 (ms) |
|------|------------|-------------|-----------|-----------|
| **RequestSizeBucket** | 362 | 1372 | 3019 | 1584 |
| **PerformanceAware** | 395 | 1340 | 2994 | 1576 |
| **差异** | +33 | -32 | -25 | -8 |

**发现**:
- 两种策略的延迟差异很小 (< 10%)
- 在单 GPU 模拟环境下,策略间的性能差异不明显
- 延迟主要由 SGLang 推理时间决定,Gateway 路由开销可忽略

### 5.2 Token 生成速度

| 请求类型 | Tokens | 延迟 (ms) | Tokens/sec |
|----------|--------|-----------|------------|
| **Short** | 41 | ~380 | ~108 |
| **Medium** | 267 | ~1356 | ~197 |
| **Long** | 647 | ~3006 | ~215 |

**分析**:
- Short 请求的 token/sec 较低,因为 Prefill 阶段占比大
- Medium 和 Long 请求的 token/sec 接近,说明 Decode 阶段稳定
- 符合预期:长请求的 Decode 阶段占比大,整体吞吐量更高

### 5.3 延迟分解

基于之前的实验报告 (13-增强版完整实验报告.md):

| 阶段 | 占比 | 说明 |
|------|------|------|
| **Prefill** (KV 计算 + Bootstrap) | ~10% | 并行计算,快速完成 |
| **Decode** (Token 生成) | ~86% | 自回归生成,耗时最长 |
| **Gateway** (路由 + 合并) | ~2% | Rust 高性能,开销极小 |
| **网络传输** | ~2% | 本地环回,延迟低 |

## 6. 发现的问题

### 6.1 CLI 策略注册

**问题**: 新策略未出现在 CLI 的 `--policy` 参数可选值中

**原因**: `src/main.rs` 中的 `value_parser` 列表和 `parse_policy` 函数未更新

**解决**: 
- 更新了 3 处 `value_parser` 列表
- 添加了 3 个新策略的 `parse_policy` match arm
- 重新编译 Gateway

**状态**: ✅ 已修复

### 6.2 编译器 ICE Bug

**问题**: Rust 1.94.1 编译器在检查未使用变量时发生 ICE (Internal Compiler Error)

**错误信息**:
```
thread 'rustc' panicked at ... slice index starts at 14 but ends at 12
query stack during panic:
#0 [check_mod_deathness] checking deathness of variables
```

**临时解决**: 使用 `RUSTFLAGS='-A dead_code -A unused'` 禁用相关 lint

**状态**: ⚠️ 临时解决,建议升级 Rust 版本或报告 bug

### 6.3 Gateway 停止延迟

**问题**: `killall -9 sgl-model-gateway` 后,Gateway 进程需要较长时间才能完全停止

**影响**: 测试脚本在策略间切换时需要额外等待

**建议**: 使用更优雅的关闭信号 (SIGTERM) 或添加健康检查重试

### 7.2 关键发现

2. **性能 overhead 小**: Gateway 路由开销 < 2%,对整体延迟影响可忽略
