## 二、实验架构说明

### 2.1 PD 分离架构

```
┌─────────────────────────────────────────────────────────┐
│                    Gateway (port 3000)                   │
│              Policy: request_classification              │
└──────────────┬──────────────────────┬───────────────────┘
               │                      │
      ┌────────┴────────┐    ┌────────┴────────┐
      │   Prefill-1     │    │   Prefill-2     │
      │   port 30000    │    │   port 30001    │
      │   bootstrap 9000│    │   bootstrap 9001│
      └────────┬────────┘    └────────┬────────┘
               │                      │
               └──────────┬───────────┘
                          │ KV Cache Bootstrap
               ┌──────────┴───────────┐
      ┌────────┴────────┐    ┌────────┴────────┐
      │   Decode-1      │    │   Decode-2      │
      │   port 30010    │    │   port 30011    │
      └─────────────────┘    └─────────────────┘
```

### 2.2 关键参数说明

| 参数 | 值 | 说明 |
|------|-----|------|
| `--pd prefill` | - | Prefill 节点标识 |
| `--pd decode` | - | Decode 节点标识 |
| `--mem-fraction-static` | 0.15 | 每个实例占用 15% GPU 内存 (~2.4GB) |
| `--context-length` | 2048 | 上下文长度限制 |
| `--disaggregation-bootstrap-port` | 9000/9001 | KV Cache 传输端口 |

### 2.3 GPU 内存分配

| 组件 | 内存占用 | 说明 |
|------|----------|------|
| Prefill-1 | ~2.4GB | 15% x 16GB |
| Prefill-2 | ~2.4GB | 15% x 16GB |
| Decode-1 | ~4.2GB | 15% x 16GB (实际更多) |
| Decode-2 | ~4.2GB | 15% x 16GB (实际更多) |
| **总计** | **~13.2GB** | 剩余 ~2.8GB |

## 三、实验操作步骤

### 步骤 1: 清理环境 (必须!)

旧的进程会占用 GPU 内存,导致新实例启动失败或 OOM。

```bash
# 清理所有相关进程

killall -9 sgl-model-gateway 2>/dev/null || true

# 等待 3 秒确保端口释放
sleep 3

# 验证清理结果
echo '=== 残留进程检查 ==='
ps aux | grep -E '(sglang|sgl-model-gateway)' | grep -v grep || echo '无残留进程'

echo '=== GPU 内存状态 ==='
nvidia-smi --query-gpu=memory.used,memory.free --format=csv,noheader
```

**观察要点**:
- 无 python3 或 sgl-model-gateway 进程
- GPU 已用内存 < 2GB (基础开销)
- GPU 可用内存 > 14GB

### 步骤 2: 启动 PD 环境

```bash

bash start-multi-pd.sh 2>&1 | tee /tmp/pd-startup.log
```

**观察要点**:

1. **启动日志检查** - 应看到以下成功标志:

   ✓ prefill-2 ready (port 30001)
   ✓ decode-1 ready (port 30010)

2. **GPU 内存检查**

3. **进程检查**

   > 这是正常的,不代表失败。使用 `-w "%{http_code}"` 显示状态码来判断。

### 步骤 3: 启动 Gateway

```bash

    --pd-disaggregation \
    --prefill http://127.0.0.1:30000 9000 \
    --prefill http://127.0.0.1:30001 9001 \
    --decode http://127.0.0.1:30010 \
    --decode http://127.0.0.1:30011 \
    --host 127.0.0.1 \
    --port 3000 \
    --policy request_classification \
    --log-level info \
    > /tmp/sgl-gateway-test.log 2>&1 &

echo 'Waiting for Gateway...'
for i in $(seq 1 30); do

  if [ "$code" = "200" ]; then
    echo "Gateway ready! (HTTP $code)"
    break
  fi
  sleep 1
done

# 额外等待 Worker 发现
sleep 5
```

## 四、故障排查指南

### 问题 1: Prefill/Decode 启动失败

**症状**: `wait_for_server` 超时

**排查步骤**:
```bash
# 1. 检查日志
tail -50 /tmp/sglang-prefill-1.log | grep -E "(error|Error|OOM|CUDA)"

# 2. 检查 GPU 内存
nvidia-smi

# 3. 检查端口占用
ss -tlnp | grep -E "30000|30001|30010|30011"
```

**常见原因**:
- GPU 内存不足 -> 清理旧进程
- 端口被占用 -> `kill` 占用进程
- 模型路径错误 -> 检查 `MODEL_PATH`

### 问题 2: Gateway 启动失败

**症状**: `curl http://127.0.0.1:3000/health` 无响应

**排查步骤**:
```bash
# 1. 检查 Gateway 日志
tail -100 /tmp/sgl-gateway-test.log | grep -E "(ERROR|WARN|fail)"

# 2. 检查 Worker 注册
grep "Activated.*worker" /tmp/sgl-gateway-test.log | wc -l

# 3. 检查 Prefill/Decode 健康状态
for port in 30000 30001 30010 30011; do
  code=$(curl -s -o /dev/null -w "%{http_code}" --max-time 5 http://127.0.0.1:$port/health)
  echo "Port $port: HTTP $code"
done
```

### 问题 3: 健康检查输出为空

**症状**: `curl http://127.0.0.1:30000/health` 没有任何输出

**原因**: SGLang 的 `/health` 端点返回 HTTP 200 + 空 body (`content-length: 0`),
这是正常行为,不代表服务异常。

**验证方法**:
```bash
# 方法 1: 查看 HTTP 状态码
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://127.0.0.1:30000/health

# 方法 2: 使用 -v 看完整响应头
curl -v http://127.0.0.1:30000/health 2>&1 | grep "< HTTP"
# 预期输出: < HTTP/1.1 200 OK

# 方法 3: 用退出码判断
curl -sf http://127.0.0.1:30000/health && echo "OK" || echo "FAIL"
# 注意: -f 仅在 HTTP 4xx/5xx 时返回失败,200 空 body 返回 OK
```

### 问题 4: 请求超时

**症状**: curl 返回超时或无响应

**排查步骤**:
```bash
# 1. 检查 Gateway 日志中的请求处理
tail -50 /tmp/sgl-gateway-test.log | grep "http_request"

# 2. 检查 SGLang 服务器是否仍在运行
ps aux | grep sglang | grep -v grep

# 3. 直接测试 SGLang 服务 (绕过 Gateway)
curl -s -X POST http://127.0.0.1:30000/generate \
  -H "Content-Type: application/json" \
  -d '{"text":"Hello","sampling_params":{"max_new_tokens":5}}'
```

### 问题 5: 测试脚本卡住

**症状**: 脚本执行到某一步后无响应

**排查步骤**:
```bash
# 1. 检查进程状态
ps aux | grep -E "(python3|sgl-model-gateway)" | grep -v grep

# 2. 检查 Gateway 日志最后几行
tail -20 /tmp/sgl-gateway-test.log

# 3. 检查 SGLang 日志
tail -20 /tmp/sglang-prefill-1.log
tail -20 /tmp/sglang-decode-1.log
```

**常见原因**:
- PD 分离 bootstrap 传输慢 -> 等待更长时间
- 进程崩溃 -> 检查日志,重启

## 七、PD 端到端测试工具 (pd-test.py)

### 7.1 工具简介

`pd-test.py` 是一个 Python 编写的端到端测试工具，用于：

1. **发送测试请求** 到 Gateway
2. **自动关联日志** 从 Gateway、Prefill 和 Decode worker
3. **显示完整请求流程** 包括路由决策、批处理信息、采样参数等
4. **验证 PD 分离架构** 是否正常工作

**核心特性**:
- ✅ 自动发现日志文件
- ✅ 智能时间窗口关联
- ✅ Bootstrap Room 映射 Prefill ↔ Decode
- ✅ 友好的 Worker 名称显示（如 `prefill-1`、`decode-2`）
- ✅ 详细的批处理和采样信息

### 7.2 前置依赖

#### 日志文件位置

工具默认从以下位置读取日志：

| 组件 | 默认路径 | 说明 |
|------|----------|------|
| **Gateway** | `/tmp/sgl-gateway-test.log` | Gateway 运行时日志 |
| **Prefill** | `/tmp/sglang-prefill-*.log` | 支持多个 Prefill 日志 |
| **Decode** | `/tmp/sglang-decode-*.log` | 支持多个 Decode 日志 |
| **启动日志** | `/tmp/pd-startup.log` | 用于解析 URL → Worker 名称映射 |

**重要**: 
- Gateway 必须以 `--log-level info` 启动，才能输出 `[PD-ROUTER]` 路由日志
- 启动日志必须由 `start-multi-pd.sh` 生成（使用 `tee /tmp/pd-startup.log`）

#### 日志生成命令

```bash
# 1. 启动 PD 环境（生成启动日志）
cd /mnt/e/dev/sglang/sgl-model-gateway/model_deploy
bash start-multi-pd.sh 2>&1 | tee /tmp/pd-startup.log

# 2. 启动 Gateway（生成运行时日志）
cd /mnt/e/dev/sglang/sgl-model-gateway
./target/release/sgl-model-gateway \
    --pd-disaggregation \
    --prefill http://127.0.0.1:30000 9000 \
    --prefill http://127.0.0.1:30001 9001 \
    --decode http://127.0.0.1:30010 \
    --decode http://127.0.0.1:30011 \
    --host 127.0.0.1 \
    --port 3000 \
    --policy request_classification \
    --log-level info \
    > /tmp/sgl-gateway-test.log 2>&1 &
```

### 7.4 工作原理

#### 日志关联机制

工具使用三种方式关联日志：

1. **请求 ID 匹配** (优先)
   - 通过 `X-Request-ID` HTTP header 传递唯一请求 ID
   - Gateway 日志记录 `request_id`
   - SGLang 日志记录 `rid` (Request ID)
   - 精确匹配，无误判

2. **Bootstrap Room 映射** (辅助)
   - Gateway 为每个请求生成唯一的 `bootstrap_room` ID
   - Prefill 和 Decode 日志都包含相同的 `bootstrap_room`
   - 用于验证 Prefill → Decode 的 KV Cache 传输

3. **时间窗口关联** (降级)
   - 当无法通过 ID 匹配时，使用时间窗口（默认 60 秒）
   - 自动选择最接近请求时间的日志条目
   - 过滤掉无关的历史日志

#### URL → Worker 名称映射

工具从启动日志 `/tmp/pd-startup.log` 中解析映射关系：

```
Prefill-1: http://127.0.0.1:30000 (bootstrap: 9000)
Prefill-2: http://127.0.0.1:30001 (bootstrap: 9001)
Decode-1:  http://127.0.0.1:30010
Decode-2:  http://127.0.0.1:30011
```

解析后，输出中会显示友好的名称（如 `prefill-1`）而非原始 URL。

#### 时间处理

- **Gateway 日志**: 使用 UTC 时间
- **SGLang 日志**: 使用本地时间（CST/UTC+8）
- 工具自动进行时区转换，确保时间比较准确

### 7.5 故障排查

#### 问题 1: "No SGLang log entries found"

**原因**: 
- 日志文件路径不正确
- 日志未刷新到磁盘

**解决**:
```bash
# 检查日志文件是否存在
ls -la /tmp/sglang-*.log

# 手动刷新日志（如果有缓冲）
sync

# 指定正确的日志目录
python3 pd-test.py "Hello" --log-dir /tmp
```

#### 问题 2: "No Gateway log entry found"

**原因**:
- Gateway 未启动或未输出日志
- 日志级别设置不正确

**解决**:
```bash
# 检查 Gateway 是否运行
ps aux | grep sgl-model-gateway

# 检查日志文件
tail -20 /tmp/sgl-gateway-test.log

# 确保使用 --log-level info 启动 Gateway
```

#### 问题 3: 显示了太多日志条目

**原因**: 时间窗口过大，匹配到多个请求

**解决**:
```bash
# 减小时间窗口（秒）
python3 pd-test.py "Hello" --window 10
```

#### 问题 4: Worker 名称显示为 URL

**原因**: 未找到启动日志 `/tmp/pd-startup.log`

**解决**:
```bash
# 检查启动日志是否存在
cat /tmp/pd-startup.log

# 如果不存在，重新启动 PD 环境并保存日志
bash start-multi-pd.sh 2>&1 | tee /tmp/pd-startup.log
```

## 八、常见问题 FAQ

**Q1: `curl /health` 没有输出,是不是服务挂了?**

A: 不是。SGLang 的 `/health` 端点返回 HTTP 200 + 空 body,所以 `curl` 不打印任何内容是正常的。用以下方式验证:
```bash
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://127.0.0.1:30000/health
# 输出 "HTTP 200" 表示正常
```

**Q2: GPU 内存不够怎么办?**

A:
1. 确保清理所有旧进程 (`killall -9 python3`)
2. 关闭其他 GPU 占用程序
3. 降低 `mem-fraction-static` (如 0.12)
4. 减少实例数量 (1P+1D)

**Q3: 请求延迟不稳定?**

A: PD 分离架构中,首次请求需要 bootstrap 传输,延迟较高。后续请求会利用缓存,延迟降低。建议:
- 发送 warmup 请求预热
- 取 3 次平均值
- 忽略首次请求

**Q4: 如何测试其他策略?**

A: 修改 Gateway 启动参数中的 `--policy`:
```bash
--policy request_size_bucket   # 测试 RequestSizeBucket
--policy performance_aware     # 测试 PerformanceAware
--policy request_classification # 测试 RequestClassification
```
