## 二、实验架构说明

### 2.1 PD 分离架构

```
┌─────────────────────────────────────────────────────────┐
│                    Gateway (port 3000)                   │
│              Policy: request_classification              │
└──────────────┬──────────────────────┬───────────────────┘
               │                      │
      ┌────────┴────────┐    ┌────────┴────────┐
      │   Prefill-1     │    │   Prefill-2     │
      │   port 30000    │    │   port 30001    │
      │   bootstrap 9000│    │   bootstrap 9001│
      └────────┬────────┘    └────────┬────────┘
               │                      │
               └──────────┬───────────┘
                          │ KV Cache Bootstrap
               ┌──────────┴───────────┐
      ┌────────┴────────┐    ┌────────┴────────┐
      │   Decode-1      │    │   Decode-2      │
      │   port 30010    │    │   port 30011    │
      └─────────────────┘    └─────────────────┘
```

### 2.2 关键参数说明

| 参数 | 值 | 说明 |
|------|-----|------|
| `--pd prefill` | - | Prefill 节点标识 |
| `--pd decode` | - | Decode 节点标识 |
| `--mem-fraction-static` | 0.15 | 每个实例占用 15% GPU 内存 (~2.4GB) |
| `--context-length` | 2048 | 上下文长度限制 |
| `--disaggregation-bootstrap-port` | 9000/9001 | KV Cache 传输端口 |

### 2.3 GPU 内存分配

| 组件 | 内存占用 | 说明 |
|------|----------|------|
| Prefill-1 | ~2.4GB | 15% × 16GB |
| Prefill-2 | ~2.4GB | 15% × 16GB |
| Decode-1 | ~4.2GB | 15% × 16GB (实际更多) |
| Decode-2 | ~4.2GB | 15% × 16GB (实际更多) |
| **总计** | **~13.2GB** | 剩余 ~2.8GB |

## 三、实验操作步骤

### 步骤 1: 清理环境 (必须!)

**重要性**: ⭐⭐⭐⭐⭐

旧的进程会占用 GPU 内存,导致新实例启动失败或 OOM。

```bash
# 在 WSL 中执行
wsl bash -c "
  # 清理所有相关进程
  killall -9 python3 2>/dev/null || true
  killall -9 sgl-model-gateway 2>/dev/null || true
  
  # 等待 3 秒确保端口释放
  sleep 3
  
  # 验证清理结果
  echo '=== 残留进程检查 ==='
  ps aux | grep -E '(sglang|sgl-model-gateway)' | grep -v grep || echo '无残留进程'
  
  echo '=== GPU 内存状态 ==='
  nvidia-smi --query-gpu=memory.used,memory.free --format=csv,noheader
"
```

**观察要点**:
- ✅ 无 python3 或 sgl-model-gateway 进程
- ✅ GPU 已用内存 < 2GB (基础开销)
- ✅ GPU 可用内存 > 14GB

## 四、故障排查指南

### 问题 1: Prefill/Decode 启动失败

**症状**: `wait_for_server` 超时

**排查步骤**:
```bash
# 1. 检查日志
wsl tail -50 /tmp/sglang-prefill-1.log | grep -E "(error|Error|OOM|CUDA)"

# 2. 检查 GPU 内存
wsl nvidia-smi

# 3. 检查端口占用
wsl netstat -tlnp | grep -E "30000|30001|30010|30011"
```

**常见原因**:
- GPU 内存不足 → 清理旧进程
- 端口被占用 → `kill` 占用进程
- 模型路径错误 → 检查 `MODEL_PATH`

### 问题 2: Gateway 启动失败

**症状**: `curl http://127.0.0.1:3000/health` 超时

**排查步骤**:
```bash
# 1. 检查 Gateway 日志
wsl tail -100 /tmp/sgl-gateway-test.log | grep -E "(ERROR|WARN|fail)"

# 2. 检查 Worker 注册
wsl grep "Activated.*worker" /tmp/sgl-gateway-test.log | wc -l

# 3. 检查 Prefill/Decode 健康状态
wsl curl -sf http://127.0.0.1:30000/health && echo "P1 OK" || echo "P1 FAIL"
wsl curl -sf http://127.0.0.1:30001/health && echo "P2 OK" || echo "P2 FAIL"
wsl curl -sf http://127.0.0.1:30010/health && echo "D1 OK" || echo "D1 FAIL"
wsl curl -sf http://127.0.0.1:30011/health && echo "D2 OK" || echo "D2 FAIL"
```

### 问题 3: 请求超时

**症状**: curl 返回超时或无响应

**排查步骤**:
```bash
# 1. 检查是否是 Windows/WSL 互操作问题
# 必须在 WSL 内部执行 curl
wsl bash -c "curl -sf http://127.0.0.1:3000/health"

# 2. 检查 Gateway 日志中的请求处理
wsl tail -50 /tmp/sgl-gateway-test.log | grep "http_request"

# 3. 检查 SGLang 服务器是否仍在运行
wsl ps aux | grep sglang | grep -v grep
```

**解决方案**:
- 创建 WSL 内部测试脚本 (参考步骤 4)
- 避免从 Windows 直接调用 WSL 的 curl

### 问题 4: 测试脚本卡住

**症状**: 脚本执行到某一步后无响应

**排查步骤**:
```bash
# 1. 检查进程状态
wsl ps aux | grep -E "(python3|sgl-model-gateway)" | grep -v grep

# 2. 检查 Gateway 日志最后几行
wsl tail -20 /tmp/sgl-gateway-test.log

# 3. 检查 SGLang 日志
wsl tail -20 /tmp/sglang-prefill-1.log
wsl tail -20 /tmp/sglang-decode-1.log
```

**常见原因**:
- PD 分离 bootstrap 传输慢 → 等待更长时间
- 进程崩溃 → 检查日志,重启

## 八、常见问题 FAQ

**Q1: 为什么必须在 WSL 内部执行 curl?**

A: Windows 调用 WSL 时存在网络栈互操作问题,会返回 `WSL/Service/0x8007274c` 错误。必须在 WSL 内部执行网络请求。

**Q2: GPU 内存不够怎么办?**

A: 
1. 确保清理所有旧进程 (`killall -9 python3`)
2. 关闭其他 GPU 占用程序
3. 降低 `mem-fraction-static` (如 0.12)
4. 减少实例数量 (1P+1D)

**Q3: 请求延迟不稳定?**

A: PD 分离架构中,首次请求需要 bootstrap 传输,延迟较高。后续请求会利用缓存,延迟降低。建议:
- 发送 warmup 请求预热
- 取 3 次平均值
- 忽略首次请求

**Q4: 如何测试其他策略?**

A: 修改 Gateway 启动参数中的 `--policy`:
```bash
--policy request_size_bucket   # 测试 RequestSizeBucket
--policy performance_aware     # 测试 PerformanceAware
--policy request_classification # 测试 RequestClassification
```
