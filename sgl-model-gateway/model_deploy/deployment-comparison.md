# 部署方案对比与选择指南

## 📊 方案对比总览

| 特性 | 单机部署 (现有) | Kubernetes + Hami vGPU (新增) |
|------|----------------|------------------------------|
| **适用场景** | 开发测试、个人使用 | 生产环境、多用户共享 |
| **部署复杂度** | ⭐ 简单 | ⭐⭐⭐⭐ 较复杂 |
| **资源利用率** | ⭐⭐ 低 (单模型占满) | ⭐⭐⭐⭐⭐ 高 (多实例共享) |
| **并发能力** | ⭐⭐ 有限 | ⭐⭐⭐⭐ 强 (多实例并行) |
| **扩展性** | ⭐ 差 | ⭐⭐⭐⭐⭐ 优秀 |
| **成本** | 低 | 中等 (需要 K8s 集群) |
| **维护成本** | 低 | 中等 |

---

## 🎯 方案一：单机部署 (当前方案)

### 架构图

```
┌─────────────────────────────┐
│   Windows 11 + WSL2         │
│                             │
│   ┌─────────────────────┐   │
│   │ SGLang Server       │   │
│   │ - Prefill + Decode  │   │
│   │ - Unified Schedule  │   │
│   │                     │   │
│   │ Port: 30000         │   │
│   └─────────────────────┘   │
│                             │
│   GPU: 4070 Ti Super        │
│   VRAM: 16GB (独占)         │
└─────────────────────────────┘
```

### 优势
- ✅ **简单易用**: 一行命令启动
- ✅ **快速上手**: 无需 Kubernetes 知识
- ✅ **资源开销小**: 无容器编排 overhead
- ✅ **适合开发**: 调试方便，问题易排查

### 劣势
- ❌ **资源浪费**: 单模型独占 16GB 显存
- ❌ **无法扩展**: 只能运行一个实例
- ❌ **并发受限**: 统一调度效率较低
- ❌ **生产限制**: 缺乏高可用、负载均衡

### 最佳实践场景
1. **个人开发测试**
2. **原型验证**
3. **小规模演示**
4. **学习研究**

### 性能指标 (实测)
- 显存占用：~10.9GB (68%)
- 并发请求：~8-12 个
- 吞吐量：~60 tokens/s (单用户)

## 🚀 方案二：Kubernetes + Hami vGPU + PD 分离

### 架构图

```
┌───────────────────────────────────────────┐
│   Kubernetes Cluster                      │
│                                           │
│   ┌─────────────────────────────────┐    │
│   │  Node: 4070 Ti Super (16GB)     │    │
│   │                                 │    │
│   │  ┌───────┐ ┌───────┐ ┌───────┐ │    │
│   │  │vGPU 1 │ │vGPU 2 │ │vGPU 3 │ │    │
│   │  │ 6GB   │ │ 6GB   │ │ 4GB   │ │    │
│   │  │Prefill│ │Decode │ │Router │ │    │
│   │  └───────┘ └───────┘ └───────┘ │    │
│   └─────────────────────────────────┘    │
└───────────────────────────────────────────┘
```

### 优势
- ✅ **资源高效**: 一卡多用，显存利用率 >85%
- ✅ **PD 分离**: Prefill 和 Decode 独立优化
- ✅ **弹性扩展**: 支持水平扩缩容
- ✅ **生产就绪**: 高可用、负载均衡、自动故障恢复
- ✅ **多租户**: 支持多个团队/项目共享 GPU

### 劣势
- ❌ **复杂度高**: 需要 Kubernetes 知识
- ❌ **部署成本高**: 需要 K8s 集群环境
- ❌ **学习曲线**: 概念多，工具链复杂
- ❌ **资源开销**: K8s 组件占用一定资源

### 最佳实践场景
1. **生产环境部署**
2. **多用户共享服务**
3. **企业级 API 服务**
4. **需要高可用的场景**
5. **资源受限但需多实例**

### 预期性能指标
- 显存利用率：~85-90%
- 并发请求：~30-50 个 (多实例合计)
- 吞吐量：~250-300 tokens/s (aggregate)
- TTFT: <500ms (P95)

## 🤔 如何选择？

### 选择单机部署，如果：

- [x] 你只需要自己使用
- [x] 主要用于开发测试
- [x] 不想学习 Kubernetes
- [x] 追求快速部署
- [x] 不需要高可用性

### 选择 Kubernetes + Hami vGPU，如果：

- [x] 需要支持多用户/多团队
- [x] 要部署到生产环境
- [x] 需要高可用和负载均衡
- [x] 希望最大化 GPU 利用率
- [x] 未来可能需要扩展

## 📈 迁移路径

### 从单机到 K8s 的演进路线

```
阶段 1: 单机部署
└─> 验证模型效果
    └─> 收集用户需求
        └─> 性能基准测试

阶段 2: Docker 容器化
└─> 打包为 Docker 镜像
    └─> 使用 docker-compose 管理
        └─> 基础监控告警

阶段 3: Kubernetes 部署
└─> 搭建 K8s 集群
    └─> 安装 Hami vGPU
        └─> 部署 PD 分离架构
            └─> 配置监控和自动伸缩

阶段 4: 生产优化
└─> 多节点集群
    └─> 自动扩缩容 (HPA/VPA)
        └─> 多副本高可用
            └─> 完善的监控体系
```

## 🔧 混合部署策略

也可以采用渐进式方案：

### 方案 A: 开发 + 生产分离

```
开发环境：单机部署
  - 快速迭代
  - 功能验证
  
生产环境：K8s + Hami
  - 稳定可靠
  - 多用户服务
```

### 方案 B: 双轨制

```
轻量应用：单机部署
  - 内部工具
  - 小流量场景

核心应用：K8s + Hami
  - 对外 API
  - 高并发场景
```

|------|------|

- 资源利用率提升：~2-3x
- 并发能力提升：~3-4x
- 运维自动化：长期节省时间

## 📝 总结

| 维度 | 推荐方案 | 理由 |
|------|---------|------|
| **个人学习** | 单机部署 | 简单快速，成本低 |
| **团队开发** | 单机 → K8s | 渐进式升级 |
| **生产环境** | K8s + Hami | 高可用、可扩展 |
| **企业服务** | K8s + Hami | 多租户、资源隔离 |
| **科研机构** | K8s + Hami | 资源共享、高效利用 |

**核心建议**: 
- 从简单开始，根据需求逐步升级
- 不要过度设计，够用就好
- 预留扩展接口，方便未来升级
